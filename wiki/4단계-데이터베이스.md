# 4단계: 데이터베이스

> FE도 쿼리를 **읽고 인덱스 이슈를 지적**할 수 있어야 한다. N+1 문제를 GraphQL에서 마주쳤을 때 당황하지 않는 수준이 목표다.

**학습 기간**: 0.5주

* * *

## 1. DB 기초와 선택 기준

### 1-1. RDB vs NoSQL

| | RDB (PostgreSQL, MySQL) | NoSQL |
|---|---|---|
| 스키마 | 엄격 | 유연 |
| 트랜잭션 | ACID | 대부분 BASE |
| 조인 | 강함 | 약함 |
| 스케일 | Vertical 위주 | Horizontal 쉬움 |

### 1-2. NoSQL 분류
- **Document**: MongoDB (JSON 유사)
- **Key-Value**: Redis, DynamoDB
- **Column-family**: Cassandra
- **Graph**: Neo4j

### 1-3. 선택 기준
- 스키마가 자주 바뀌면 Document를 쓴다
- 복잡한 조인이 많으면 RDB를 쓴다
- 극단적 처리량이 필요하면 Key-Value를 쓴다
- 관계가 복잡하면 Graph를 쓴다

## 2. SQL 기본기

### 2-1. 기본 문법
```sql
SELECT id, name
FROM users
WHERE age >= 20
ORDER BY created_at DESC
LIMIT 10;
```

### 2-2. JOIN
- INNER JOIN: 양쪽에 모두 존재하는 행만
- LEFT JOIN: 왼쪽 전부 + 오른쪽 매칭
- RIGHT JOIN: 그 반대
- FULL OUTER JOIN: 양쪽 전부
- CROSS JOIN: 곱집합 (거의 쓰지 않는다)

### 2-3. 집계와 그룹
```sql
SELECT category, COUNT(*), AVG(price)
FROM products
GROUP BY category
HAVING COUNT(*) > 10;
```
- `WHERE`는 그룹 전, `HAVING`은 그룹 후에 적용한다

### 2-4. 서브쿼리
- 스칼라 서브쿼리, 인라인 뷰, EXISTS를 구분한다
- 대부분은 JOIN으로 재작성할 수 있다

### 2-5. EXPLAIN으로 실행 계획 읽기
- Seq Scan / Index Scan / Index Only Scan을 구분한다
- Nested Loop / Hash Join / Merge Join을 구분한다
- 비용(cost)과 실제 시간(actual)을 함께 본다

## 3. 정규화

### 3-1. 이상(Anomaly) 현상
- 삽입 이상, 갱신 이상, 삭제 이상
- 중복이 일관성을 깬다

### 3-2. 1NF / 2NF / 3NF
- **1NF**: 원자값 (배열·리스트 금지)
- **2NF**: 1NF + 부분 함수 종속 제거 (복합 키 일부에만 종속되면 안 됨)
- **3NF**: 2NF + 이행 종속 제거 (A → B → C 구조 금지)
- BCNF, 4NF, 5NF는 이름만 알면 된다

### 3-3. 비정규화 (Denormalization)
- 읽기 성능을 위해 일부러 중복을 허용한다
- 집계 테이블, 카운트 캐시 컬럼 등
- **"언제 정규화를 깨야 하는지 아는 것이 진짜 실력"**이라는 점을 기억한다

* * *

## 4. 인덱스

### 4-1. B-Tree 인덱스 (가장 흔한 형태)
- 균형 트리, 노드당 여러 자식 → 디스크 I/O를 최소화한다
- 조회·삽입·삭제 모두 O(log n)
- 정렬 상태가 유지되므로 범위 검색에 강하다

### 4-2. 인덱스를 못 타는 경우
```sql
WHERE name LIKE '%abc'      -- 접두사 없음
WHERE UPPER(name) = 'ABC'   -- 함수 적용
WHERE age + 1 = 20          -- 컬럼 연산
WHERE name != 'abc'         -- 부정 조건
```
→ 함수 기반 인덱스(`UPPER(name)` 인덱스)로 해결할 수 있다

### 4-3. Composite Index
- `(a, b, c)` 인덱스는 `a`, `(a, b)`, `(a, b, c)` 쿼리에 유효하다
- `b` 단독이나 `(b, c)` 쿼리에는 쓸 수 없다 → **선두 컬럼 규칙**

### 4-4. Hash Index
- 등가 비교만 가능하고 범위 검색에는 쓸 수 없다
- PostgreSQL에서 제한적으로 사용한다

### 4-5. Covering Index
- 필요한 컬럼이 전부 인덱스에 포함되면 테이블 접근 없이 답한다

* * *

## 5. 트랜잭션

### 5-1. ACID
- **Atomicity**: 전부 되거나 전부 안 된다
- **Consistency**: 제약 조건을 유지한다
- **Isolation**: 동시 실행 트랜잭션끼리 서로를 간섭하지 않는다
- **Durability**: 커밋되면 영구 저장된다

### 5-2. Commit / Rollback
- `BEGIN` 또는 `START TRANSACTION`으로 명시적으로 시작한다
- `COMMIT` 또는 `ROLLBACK`으로 종료한다
- Savepoint로 부분 롤백을 할 수 있다

### 5-3. 격리 수준과 이상 현상

| 격리 수준 | Dirty Read | Non-repeatable | Phantom |
|---|---|---|---|
| Read Uncommitted | O | O | O |
| Read Committed | X | O | O |
| Repeatable Read | X | X | O (MySQL은 X) |
| Serializable | X | X | X |

- **Dirty Read**: 커밋되지 않은 데이터를 읽음
- **Non-repeatable Read**: 같은 쿼리 두 번에 결과가 다름
- **Phantom Read**: 범위 쿼리 결과의 행 수가 달라짐

### 5-4. 낙관적 락 vs 비관적 락
- 낙관적(Optimistic): 버전 컬럼으로 충돌을 감지하고 재시도한다
  - FE의 "Optimistic UI 업데이트"와 사상이 같다
- 비관적(Pessimistic): `SELECT ... FOR UPDATE`로 선점한다

* * *

## 6. FE가 자주 마주치는 이슈

### 6-1. N+1 쿼리 문제
```
SELECT * FROM posts;              -- 1번
-- 각 post마다:
SELECT * FROM users WHERE id = ?; -- N번
```
**해결**: JOIN, IN 쿼리, Dataloader 패턴(GraphQL) 중 하나를 쓴다

### 6-2. 페이지네이션
- **Offset 기반** (`LIMIT 10 OFFSET 100`): 구현이 쉽지만 깊어질수록 느리다
- **Cursor 기반**: `WHERE id < ? ORDER BY id DESC LIMIT 10`, 무한 스크롤에 적합하다

### 6-3. 소프트 삭제
- `deleted_at` 컬럼으로 논리 삭제한다
- 모든 쿼리에 `WHERE deleted_at IS NULL`을 붙여야 한다 → 누락 시 버그를 유발한다

### 6-4. 동시성 시나리오
- 좋아요 수 증가 — Race Condition
- 재고 차감 — 초과 판매 방지
- 해결: 트랜잭션 + 락 또는 atomic counter(Redis INCR)

## 연습 문제

1. `users`와 `posts` 테이블을 3NF까지 설계하라. 각 컬럼에 왜 그 타입을 썼는지 설명하라.
2. `WHERE name LIKE 'abc%'`는 인덱스를 타지만 `WHERE name LIKE '%abc'`는 못 탄다. B-Tree 구조로 이유를 설명하라.
3. N+1 쿼리 예시를 SQL로 작성하고, JOIN과 IN 방식으로 각각 해결하라.
4. Read Committed 수준에서 발생할 수 있는 이상 현상을 시나리오로 작성하라.
5. 댓글 수 표시 기능을 구현할 때, 정규화된 방식과 비정규화(카운트 컬럼) 방식의 장단점을 논하라.
