# 6단계: 소프트웨어 공학

> 시니어 면접에서 전공자와 가장 크게 벌어지는 영역이다. **패턴 이름으로 리뷰 코멘트를 쓰고, 원칙 이름으로 구조적 결정을 내릴 수 있는 수준**이 목표다. 절대 스킵하지 않는다.

**학습 기간**: 1.5주

* * *

## 1. 디자인 패턴

패턴 이름을 들었을 때 **내 코드에서 본 예시**를 댈 수 있어야 한다.

### 1-1. Observer / Pub-Sub
- 주체(Subject)가 구독자(Observer)들에게 변경을 통지한다
- **FE 실전**:
  - DOM 이벤트 리스너
  - Redux `store.subscribe()`
  - RxJS Observable
  - Vue 반응성 내부

### 1-2. Singleton
- 인스턴스 1개만 존재함을 보장한다
- **조심해야 할 패턴**이다. 전역 상태는 테스트를 어렵게 만든다
- FE에서는 ES 모듈 자체가 싱글톤이다 (import 시 1번만 평가)

### 1-3. Factory
- 인스턴스 생성을 별도 함수/클래스로 위임한다
- **FE 실전**: API 클라이언트 생성, 컴포넌트 생성 추상화

### 1-4. Strategy
- 알고리즘/정책을 교체 가능한 객체로 캡슐화한다
- **FE 실전**: 정렬 방식 선택, 폼 검증 규칙 교체, 결제 수단별 처리

### 1-5. Command
- 요청을 객체로 캡슐화한다
- **FE 실전**: Redux Action, Undo/Redo 스택, Command 팔레트

### 1-6. Proxy
- 객체 접근을 가로채 부가 동작을 삽입한다
- **FE 실전**:
  - Vue 3 반응형 (`Proxy` 직접 사용)
  - MobX
  - API mocking

### 1-7. Decorator
- 객체 기능을 **동적으로 확장**한다
- **FE 실전**: HOC (Higher-Order Component), Express 미들웨어

### 1-8. Adapter / Facade
- Adapter: 호환되지 않는 인터페이스끼리 연결한다
- Facade: 복잡한 서브시스템을 단순한 API로 래핑한다
- **FE 실전**: 서드파티 SDK 래핑, fetch/axios 추상화 레이어

### 1-9. Module
- 관련 기능을 캡슐화하고 private 영역을 유지한다
- **FE 실전**: IIFE(과거), ESM

### 1-10. State 패턴과 FSM
- 상태에 따라 다른 동작을 한다
- XState로 명시적으로 구현한다 (5단계에서 다룸)

## 2. 설계 원칙

### 2-1. SOLID

#### S — Single Responsibility Principle
- 하나의 모듈/함수는 **한 가지 이유로만 변경**되어야 한다
- FE 적용: 컴포넌트를 "UI 렌더 + 데이터 fetch + 비즈니스 로직"으로 섞지 않는다

#### O — Open-Closed Principle
- **확장에는 열려 있고, 변경에는 닫혀 있어야 한다**
- FE 적용: 새 결제 수단 추가 시 기존 코드를 수정하지 않고 추가한다 (Strategy)

#### L — Liskov Substitution Principle
- 자식 타입은 부모 타입을 **완전히 대체할 수 있어야** 한다
- FE 적용: Props 타입 계층 설계에 반영한다

#### I — Interface Segregation Principle
- 큰 인터페이스 하나보다 **여러 작은 인터페이스**가 낫다
- FE 적용: 100개 prop을 받는 거대 컴포넌트를 쪼갠다

#### D — Dependency Inversion Principle
- **추상에 의존하고, 구체에 의존하지 않는다**
- FE 적용: 컴포넌트가 `fetch`를 직접 호출하지 않고 훅을 통해 주입받는다
  ```tsx
  // Bad
  function UserList() {
    useEffect(() => { fetch('/users').then(...) }, []);
  }
  // Good
  function UserList({ useUsers }) {
    const users = useUsers();
  }
  ```

### 2-2. DRY (Don't Repeat Yourself)
- 반복은 동기화 부담을 만든다
- 단, **지식의 중복**에만 적용한다. **우연한 코드 유사**에는 적용하지 않는다

### 2-3. KISS (Keep It Simple, Stupid)
- 필요한 만큼만 복잡하게 만든다
- "언젠가 필요할 것 같아서" 미리 넣지 않는다

### 2-4. YAGNI (You Aren't Gonna Need It)
- 예측 설계를 하지 않는다
- 필요해졌을 때 추가한다

### 2-5. 응집도와 결합도
- **응집도**: 한 모듈 내부 요소들의 관련성 — **높을수록 좋다**
- **결합도**: 모듈 간 의존 강도 — **낮을수록 좋다**
- 목표는 높은 응집도 + 낮은 결합도다

### 2-6. Law of Demeter (최소 지식 원칙)
- "친구의 친구에게 말하지 말라"
- `a.b.c.d()` 형태는 피한다. 체인이 길면 구조 리팩토링을 고려한다

### 2-7. Composition over Inheritance
- 상속보다 합성을 택한다
- React는 상속 대신 children / render props / hooks로 재사용한다

## 3. 아키텍처 패턴

### 3-1. MVC / MVP / MVVM
- **MVC**: Model-View-Controller
- **MVP**: View와 Model을 Presenter가 중재한다 (View가 수동적)
- **MVVM**: ViewModel이 데이터 바인딩을 제공한다 → Vue, Angular

### 3-2. Flux / Redux
- 단방향 데이터 흐름을 따른다: **Action → Dispatcher → Store → View**
- 예측 가능한 상태 변경을 보장한다
- Redux는 Flux 아키텍처의 구현체다

### 3-3. 클린 아키텍처
```
┌─────────────────────────┐
│   UI (Framework)        │
├─────────────────────────┤
│   Interface Adapters    │  (Presenters, Controllers)
├─────────────────────────┤
│   Use Cases             │  (Application Logic)
├─────────────────────────┤
│   Entities (Domain)     │  (Business Rules)
└─────────────────────────┘
         의존성 방향 →
```
- 바깥은 안쪽을 알지만, 안쪽은 바깥을 모른다
- **도메인을 프레임워크에 오염시키지 않는다**

### 3-4. Hexagonal (Ports & Adapters)
- 도메인을 중심에 두고 Port(인터페이스) + Adapter(구현)로 외부와 소통한다
- 테스트 시 Adapter만 교체한다

### 3-5. FSD (Feature-Sliced Design)
FE에서 채택이 늘고 있다.
```
shared → entities → features → widgets → pages → app
```
- 아래 레이어가 위를 의존할 수 없다
- 기능 단위 응집을 강제한다

### 3-6. Container / Presentational
- **Container**: 상태·로직 담당
- **Presentational**: UI만 담당
- Hooks 시대엔 덜 강조되지만, 관심사 분리의 기본으로 기억한다

### 3-7. 훅 추출 패턴
- 비즈니스 로직을 커스텀 훅으로 분리한다
- 컴포넌트는 UI + 훅 조합으로 구성한다

### 3-8. 마이크로 프론트엔드
- 팀별로 독립 배포한다
- Module Federation (Webpack 5)
- 트레이드오프: 독립성 vs 번들 크기 중복

### 3-9. 모노레포
- Turborepo, Nx, pnpm workspace를 쓴다
- 코드 공유 + 독립 배포 + 캐시된 빌드를 동시에 얻는다

## 4. 함수형 프로그래밍

### 4-1. 순수 함수
- 같은 입력에는 같은 출력을 반환한다
- 사이드 이펙트가 없다
- **테스트, 캐싱, 병렬화가 쉽다**

### 4-2. 불변성
- 원본을 수정하지 않고 **새 객체를 반환**한다
- Redux/Zustand 업데이트의 기본이다
- `Object.freeze`, Immer 라이브러리를 활용한다

### 4-3. 고차 함수
- 함수를 인자로 받거나 반환하는 함수
- `map`, `filter`, `reduce`
- `debounce`, `throttle`도 고차 함수다

### 4-4. 커링 / 부분 적용
```js
const add = a => b => a + b;
const add5 = add(5);
add5(3); // 8
```
- React의 HOC는 커링과 구조가 유사하다

### 4-5. 함수 합성
```js
const pipe = (...fns) => x => fns.reduce((acc, fn) => fn(acc), x);
const transform = pipe(trim, toLowerCase, removeSpaces);
```

### 4-6. React와 FP
- 컴포넌트 = `(props) => UI` (순수 함수)
- State는 불변 업데이트를 유지한다
- 사이드 이펙트는 `useEffect`로 격리한다
- **React가 함수형으로 기울어진 이유**: 예측 가능성 + 시간 여행 디버깅 + Concurrent 모드

### 4-7. 모나드 맛보기
- "값을 감싸고 체이닝 가능한 컨테이너"로 정의한다
- `Promise`는 모나드다 (`then`이 bind에 해당)
- `Array.flatMap`도 모나드적 동작이다

## 5. 테스팅

### 5-1. 테스트 피라미드
```
        /\
       /E2E\         (적음, 느림, 비쌈)
      /------\
     /통합    \
    /----------\
   /  단위      \   (많음, 빠름, 쌈)
  --------------
```
- 피라미드가 아닌 **눈송이(아이스크림 콘)** 형태가 되면 위험 신호로 본다

### 5-2. TDD (Test-Driven Development)
1. **Red**: 실패하는 테스트를 먼저 쓴다
2. **Green**: 최소한의 코드로 통과시킨다
3. **Refactor**: 구조를 개선한다

### 5-3. 테스트 더블
| 용어 | 설명 |
|---|---|
| **Dummy** | 자리만 채움, 실제 사용 X |
| **Stub** | 미리 정한 값 반환 |
| **Spy** | 호출 기록 |
| **Mock** | 기대 동작 명시 + 검증 |
| **Fake** | 간단한 실제 구현 (in-memory DB) |

### 5-4. Mock의 함정
- 구현과 결합되면 리팩토링이 어려워진다
- **"무엇"을 테스트하는지 명확히 해야 한다.** 통합 지점만 모킹한다

### 5-5. 커버리지의 함정
- 100% 커버리지가 버그 없음을 뜻하지 않는다
- 엣지 케이스와 동시성 이슈는 커버리지로 잡히지 않는다
- 목표는 **"이 숫자"가 아니라 "이 변경이 깨뜨릴 것이 테스트에 잡히는가"**다

### 5-6. FE 테스트 도구
| 도구 | 레이어 |
|---|---|
| Jest / Vitest | 단위 |
| React Testing Library | 컴포넌트 (통합) |
| MSW | API 모킹 |
| Playwright / Cypress | E2E |
| Storybook + Chromatic | 시각 회귀 |

### 5-7. Testing Library 원칙
- **사용자가 하듯** 요소를 찾는다 (getByRole, getByLabelText)
- 구현 세부(className, state)를 테스트하지 않는다
- **"저항하는 테스트"가 좋은 테스트**임을 기억한다

## 6. 버전 관리 (Git)

### 6-1. Git은 DAG
- 모든 커밋은 이전 커밋을 가리키는 **방향성 비순환 그래프**다
- 병합 커밋은 부모가 여럿이다

### 6-2. 내부 오브젝트
- **blob**: 파일 내용
- **tree**: 디렉터리 구조 (blob과 tree의 목록)
- **commit**: 트리 + 부모 + 메타
- **tag**: 특정 커밋 참조

### 6-3. HEAD, Ref, Reflog
- `HEAD`: 현재 위치 포인터
- `refs/heads/main`: 브랜치
- `reflog`: HEAD가 가리켰던 기록 (실수 복구용)

### 6-4. Merge vs Rebase
- **Merge**: 히스토리를 그대로 두고 병합 커밋을 만든다
- **Rebase**: 커밋을 다른 베이스 위로 재적용한다 → 선형 히스토리
- Rule of thumb: **공유된 브랜치는 rebase 하지 않는다**

### 6-5. Reset 세 모드
- `--soft`: HEAD만 이동하고 staged는 유지
- `--mixed` (기본): HEAD + staged 되돌림, working directory 유지
- `--hard`: 전부 되돌림 (위험)

### 6-6. Cherry-pick / Revert
- Cherry-pick: 특정 커밋만 복사한다
- Revert: 이전 커밋의 반대 변경을 **새 커밋으로** 추가한다 (공유 브랜치에서 안전)

### 6-7. 브랜치 전략
- **Git Flow**: main/develop/feature/release/hotfix — 복잡하다
- **GitHub Flow**: main + feature PR — 심플하다
- **Trunk-Based**: main에 자주 머지 + 피처 플래그 — 대규모 팀에 적합하다

## 7. 배포와 CI/CD (개요)

### 7-1. CI/CD 파이프라인
```
Push → Lint → Test → Build → (Staging) → (승인) → Production
```

### 7-2. 배포 전략
- **Blue/Green**: 두 환경을 유지하고 스위치한다
- **Canary**: 일부 사용자에게 먼저 배포한다
- **Rolling**: 점진적으로 교체한다

### 7-3. 피처 플래그
- 배포와 릴리스를 분리한다
- A/B 테스트에 활용한다
- 빠른 롤백이 가능하다

### 7-4. 환경 분리
- local / dev / staging / prod
- 환경 변수(`.env`)로 분기한다

## 연습 문제

1. 최근 프로젝트에서 Observer/Pub-Sub 패턴이 쓰인 곳 3곳을 찾아라.
2. 아래 컴포넌트를 SOLID 원칙 중 위반되는 것을 기준으로 리팩토링하라.
   ```tsx
   function UserDashboard() {
     const [users, setUsers] = useState([]);
     useEffect(() => {
       fetch('/api/users').then(r => r.json()).then(data => {
         const active = data.filter(u => u.status === 'active');
         const sorted = active.sort((a, b) => b.score - a.score);
         setUsers(sorted);
       });
     }, []);
     return <div>{users.map(u => <div>{u.name}: {u.score}</div>)}</div>;
   }
   ```
3. Redux의 "단방향 데이터 흐름"과 Observer 패턴의 관계를 설명하라.
4. 클린 아키텍처에서 FE 프레임워크는 어느 레이어에 속하는가? 왜 그런가?
5. `map`, `filter`, `reduce`만으로 객체 배열에서 "활성 사용자의 평균 점수"를 구하라.
6. 다음 함수를 TDD로 구현하라: 문자열 배열을 받아 길이순 내림차순으로 정렬하되, 같은 길이는 알파벳 순으로.
7. Merge와 Rebase 중 공유 브랜치에서 Rebase가 위험한 이유를 그래프로 설명하라.
8. 테스트 커버리지 100%인데 버그가 발생할 수 있는 시나리오를 3가지 만들어라.
