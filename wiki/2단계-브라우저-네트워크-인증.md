# 2단계: 브라우저·네트워크·인증

FE 면접 최빈출 영역이다. 가장 많은 시간을 투자해야 한다. 이 단계를 끝내면 "주소창에 URL을 입력하면 어떤 일이 일어나는가"에 30분 단위로 답할 수 있어야 한다.

**기간**: 2.5주

* * *

## 1. 네트워크 기초

### 1-1. OSI 7계층과 TCP/IP 4계층
- 외우지 말고 흐름으로 이해해야 한다
- 물리 → 데이터 링크 → 네트워크(IP) → 전송(TCP/UDP) → 세션/표현/응용(HTTP)
- 각 계층이 서로를 몰라도 되는 이유는 캡슐화다

### 1-2. TCP vs UDP
| | TCP | UDP |
|---|---|---|
| 연결 | 3-way handshake | 비연결 |
| 순서 보장 | O | X |
| 혼잡 제어 | O | X |
| 용도 | HTTP, SSH | DNS, 영상 스트림, HTTP/3(QUIC) |

### 1-3. 3-way / 4-way Handshake
- SYN → SYN+ACK → ACK (연결)
- FIN → ACK → FIN → ACK (종료)
- `TIME_WAIT` 상태가 존재하는 이유를 알아야 한다

### 1-4. 흐름 제어 vs 혼잡 제어
- 흐름 제어: 수신자 버퍼 기준 (Window Size)
- 혼잡 제어: 네트워크 전체 기준 (Slow Start, AIMD)

### 1-5. DNS
- 재귀 질의 vs 반복 질의
- Root → TLD → Authoritative
- 캐싱 계층: 브라우저 → OS → ISP → 권한 서버
- TTL의 의미

* * *

## 2. HTTP와 HTTPS

DevTools Network 탭을 보고 문제를 진단할 수 있어야 한다.

### 2-1. HTTP 요청/응답 구조
```
GET /users/1 HTTP/1.1
Host: api.example.com
Authorization: Bearer xxx
Content-Type: application/json

{ "body": "..." }
```

### 2-2. 상태 코드
- 2xx: 성공 (200, 201, 204)
- 3xx: 리다이렉트 (301, 302, 304)
- 4xx: 클라이언트 오류 (400, 401, 403, 404, 409, 429)
- 5xx: 서버 오류 (500, 502, 503, 504)

### 2-3. 주요 헤더
- 요청: `Authorization`, `Content-Type`, `Accept`, `Cookie`, `Origin`, `Referer`
- 응답: `Set-Cookie`, `Cache-Control`, `ETag`, `Access-Control-*`
- 메서드: GET (안전, 멱등), POST (비멱등), PUT/DELETE (멱등), PATCH

### 2-4. HTTP/1.1 → HTTP/2 → HTTP/3
| | 1.1 | 2 | 3 |
|---|---|---|---|
| 전송 | TCP | TCP | UDP (QUIC) |
| 멀티플렉싱 | X (HOL blocking) | O | O |
| 헤더 압축 | X | HPACK | QPACK |
| 서버 푸시 | X | O (deprecated 추세) | O |

### 2-5. TLS 1.3 핸드셰이크
- ClientHello → ServerHello (+ 인증서) → Finished
- 1-RTT 연결, 재연결 시 0-RTT
- 인증서 체인 검증, SNI, HSTS

* * *

## 3. 통신 프로토콜과 API 스타일

상황에 맞는 통신 방식을 고르고 근거를 댈 수 있어야 한다.

### 3-1. REST
- 리소스 중심, HTTP 메서드로 동작 표현
- 멱등성과 안전성
- HATEOAS (실무에선 거의 무시)

### 3-2. GraphQL
- 오버페칭/언더페칭 해결
- 단일 엔드포인트
- N+1 문제는 Dataloader로 해결한다

### 3-3. WebSocket / SSE / Long Polling
- WebSocket: 양방향, TCP 위 자체 프로토콜 (101 Switching Protocols)
- SSE: 단방향 서버→클라이언트, HTTP 기반
- Long Polling: HTTP 연결 유지, 서버가 응답을 지연

### 3-4. Fetch / Axios
- `AbortController`로 요청 취소
- 인터셉터 패턴 (Axios)
- 재시도 전략 (Exponential Backoff)

* * *

## 4. 인증과 인가

토큰 기반 인증 시스템을 혼자 설계할 수 있어야 한다.

### 4-1. 인증 vs 인가
- Authentication: 누구인가 (로그인)
- Authorization: 무엇을 할 수 있는가 (권한)

### 4-2. 세션 기반 인증
- 서버가 세션 저장소에 상태 유지
- 쿠키로 세션 ID 전달
- 스케일 아웃 시 세션 공유 문제 발생 (Redis 등으로 해결)

### 4-3. JWT (JSON Web Token)
- 구조: `header.payload.signature` (Base64URL 인코딩)
- 서버는 상태를 저장하지 않는다 (stateless)
- 서명 검증으로 위변조를 막는다. 내용은 암호화되지 않으므로 민감 정보는 넣지 않아야 한다
- 만료 `exp`, 발급자 `iss`, 주체 `sub`

### 4-4. Access Token vs Refresh Token
- Access: 짧은 수명 (~15분), API 호출용
- Refresh: 긴 수명 (~2주), Access 재발급용
- Refresh Token Rotation: 사용할 때마다 새 것을 발급해 탈취를 감지한다

### 4-5. OAuth 2.0
- Authorization Code Grant (가장 안전, 서버가 있을 때)
  1. 클라이언트 → 인증 서버 (`response_type=code`)
  2. 사용자 동의 → Authorization Code 발급
  3. 클라이언트(서버) → 인증 서버에 Code + Secret → Access Token
- Implicit Grant: deprecated
- Client Credentials: 서버 간 통신
- Resource Owner Password: 공식 deprecated

### 4-6. PKCE (Proof Key for Code Exchange)
- SPA/모바일처럼 Client Secret을 숨길 수 없는 환경에서 필수
- `code_verifier` (랜덤) + `code_challenge` (SHA-256 해시)
- OAuth 2.1에서 사실상 필수

### 4-7. OIDC (OpenID Connect)
- OAuth 위에 인증 레이어를 올린 표준
- ID Token (JWT) 추가 발급 — 사용자 정보 포함
- 소셜 로그인의 실체

### 4-8. 토큰 저장 위치
| 위치 | XSS 취약 | CSRF 취약 | 비고 |
|---|---|---|---|
| LocalStorage | O | X | JS 접근 가능 → XSS에 치명적 |
| Cookie (HttpOnly) | X | O | CSRF 토큰 / SameSite로 방어 |
| Memory | X | X | 새로고침 시 소실 |

**현실적 기준**: Access Token은 메모리, Refresh Token은 HttpOnly + Secure + SameSite=Strict 쿠키에 저장한다.

## 5. 브라우저 렌더링

"URL 입력부터 렌더링까지"를 5분간 설명할 수 있어야 한다.

### 5-1. 전체 흐름
```
URL 입력
  ↓
DNS 조회
  ↓
TCP 연결 (+ TLS 핸드셰이크)
  ↓
HTTP 요청
  ↓
HTML 응답
  ↓
파싱 + 렌더링
```

### 5-2. Critical Rendering Path
1. HTML 파싱 → DOM 트리
2. CSS 파싱 → CSSOM 트리
3. DOM + CSSOM → Render Tree (보이는 것만)
4. Layout (Reflow) — 위치·크기 계산
5. Paint — 픽셀 채우기
6. Composite — 레이어 합성 (GPU)

### 5-3. Reflow vs Repaint
- Reflow: 레이아웃 재계산. 크기/위치 변경, DOM 추가/삭제, `offsetHeight` 같은 강제 동기 레이아웃에서 발생
- Repaint: 색/가시성만 변경
- Composite only: `transform`, `opacity` — 가장 저렴하다

### 5-4. 렌더링 최적화
- 애니메이션은 `transform` / `opacity`로 처리해야 한다
- `will-change`는 레이어 승격용이지만 남용하지 않는다 (메모리 비용)
- 스타일 변경은 `requestAnimationFrame`으로 묶어야 한다

### 5-5. 리소스 힌트
- `async`: 다운로드 중 파싱 계속, 완료 시 즉시 실행 (순서 보장 X)
- `defer`: 다운로드 중 파싱 계속, HTML 파싱 후 순서대로 실행
- `preload`: 이 페이지에서 곧 쓸 것
- `prefetch`: 다음 페이지에서 쓸 것
- `modulepreload`: ESM 모듈 전용

## 6. 이벤트 루프

비동기 코드의 실행 순서를 정확히 예측할 수 있어야 한다.

### 6-1. 구성 요소
- Call Stack
- Web APIs (setTimeout, fetch, DOM 이벤트)
- Callback Queue (macrotask)
- Microtask Queue (Promise, queueMicrotask, MutationObserver)
- Render

### 6-2. 실행 순서
1. 콜 스택의 동기 코드 실행
2. 콜 스택이 비면 마이크로태스크 큐를 전부 비운다
3. 렌더 기회 (필요 시)
4. 매크로태스크 큐에서 1개 실행
5. 반복

### 6-3. 예측 문제
```js
console.log(1);
setTimeout(() => console.log(2));
Promise.resolve().then(() => console.log(3));
console.log(4);
// 출력: 1, 4, 3, 2
```

### 6-4. requestAnimationFrame
- 렌더 직전에 실행 → 16.6ms(60fps) 주기
- 애니메이션은 `setTimeout` 대신 rAF를 써야 한다

### 6-5. Node.js 이벤트 루프
- 6단계: timers → pending → idle → poll → check → close
- `setImmediate` vs `setTimeout(fn, 0)`의 차이를 알아야 한다

## 7. 브라우저 저장소

| 저장소 | 용량 | 생명주기 | 서버 전송 | 접근 |
|---|---|---|---|---|
| Cookie | ~4KB | 설정 가능 | 자동 | JS + 서버 |
| LocalStorage | ~5-10MB | 영구 | X | JS만 |
| SessionStorage | ~5-10MB | 탭 종료 시 | X | JS만 |
| IndexedDB | 수백MB+ | 영구 | X | JS (비동기) |
| Cache Storage | 정책 | 영구 | X | Service Worker |

### 7-1. Cookie 속성
- `HttpOnly`: JS 접근 차단 → XSS 방어
- `Secure`: HTTPS에서만 전송
- `SameSite`: Lax (기본), Strict, None
- `Domain`, `Path`, `Max-Age` / `Expires`

## 8. 웹 보안

주요 공격 벡터와 방어를 코드 수준에서 설명할 수 있어야 한다.

### 8-1. 동일 출처 정책(SOP)과 CORS
- Origin = scheme + host + port
- Simple Request: GET/POST + 특정 헤더/Content-Type
- Preflight(OPTIONS): Simple이 아니면 사전 확인
- 서버 헤더: `Access-Control-Allow-Origin`, `-Methods`, `-Headers`, `-Credentials`
- 와일드카드(`*`)는 `credentials: 'include'`와 함께 쓸 수 없다

### 8-2. XSS
- Stored / Reflected / DOM-based
- 방어:
  - React/Vue의 기본 이스케이프 (단, `dangerouslySetInnerHTML`은 주의)
  - CSP (`script-src 'self'`)
  - Trusted Types API
  - 입력 sanitize (DOMPurify)

### 8-3. CSRF
- 피해자의 인증된 세션을 악용한다
- 방어:
  - SameSite 쿠키
  - CSRF 토큰 (서버 발급, 폼에 포함)
  - Double Submit Cookie 패턴

### 8-4. Clickjacking
- iframe으로 UI 위장
- 방어: `X-Frame-Options: DENY`, CSP `frame-ancestors`

### 8-5. 기타
- COOP / COEP: 사이트 격리 (Spectre 방어)
- SRI (Subresource Integrity): CDN 변조 감지
- Origin vs Site 차이 (SameSite 쿠키 맥락)

* * *

## 9. 캐싱과 CDN

캐시 헤더를 보고 전략을 설계·디버깅할 수 있어야 한다.

### 9-1. Cache-Control
- `no-store`: 절대 저장하지 않는다
- `no-cache`: 저장은 하되 매번 재검증
- `max-age=N`: N초간 fresh
- `s-maxage=N`: CDN/프록시 전용
- `immutable`: 재검증도 하지 않는다 (해시 파일명과 조합)
- `stale-while-revalidate=N`: stale 응답 제공 + 백그라운드 재검증

### 9-2. 조건부 요청
- `ETag` → `If-None-Match`
- `Last-Modified` → `If-Modified-Since`
- 304 Not Modified 응답

### 9-3. CDN
- 엣지 서버가 오리진 앞에서 캐싱
- 캐시 키: URL + 선택적 헤더
- 퍼지(Purge) / 무효화

### 9-4. Service Worker 캐싱 전략
- Cache First (정적 자원)
- Network First (API)
- Stale-While-Revalidate (UX + 최신성 균형)

## 연습 문제

1. `example.com` HTTPS 페이지에서 `api.example.com`으로 POST 시 Preflight 발생 여부를 조건으로 설명하라
2. JWT 서명 검증 방법. 공개키/대칭키 방식 차이
3. OAuth 2.0 Authorization Code Grant 흐름을 그리고 PKCE 추가 지점을 표시하라
4. 아래 코드의 출력을 예측하라
   ```js
   async function a() { console.log(1); await null; console.log(2); }
   a();
   Promise.resolve().then(() => console.log(3));
   console.log(4);
   ```
5. `transform: translateX(100px)`과 `left: 100px`의 렌더링 비용 차이를 CRP로 설명하라
6. Refresh Token을 LocalStorage에 저장하면 안 되는 이유를 XSS 시나리오로 설명하라

## 체크리스트

### 네트워크
- [ ] TCP 3-way handshake를 그릴 수 있다
- [ ] HTTP/2의 멀티플렉싱과 HOL Blocking을 설명할 수 있다
- [ ] HTTP/3이 UDP인 이유를 설명할 수 있다
- [ ] TLS 1.3 핸드셰이크 과정을 안다

### HTTP
- [ ] 멱등성과 안전성의 차이를 안다
- [ ] 주요 상태 코드와 헤더를 즉답할 수 있다
- [ ] CORS Preflight 발생 조건을 안다

### 인증
- [ ] OAuth 2.0 Authorization Code Grant를 그릴 수 있다
- [ ] PKCE가 왜 필요한지 안다
- [ ] OIDC와 OAuth의 차이를 안다
- [ ] JWT의 구조와 서명 검증을 설명할 수 있다
- [ ] 토큰 저장 위치별 보안 트레이드오프를 안다

### 렌더링
- [ ] Critical Rendering Path 6단계를 순서대로 말할 수 있다
- [ ] Reflow 트리거 조건을 안다
- [ ] `async` / `defer` / `preload` / `prefetch` 차이를 안다

### 이벤트 루프
- [ ] 마이크로태스크와 매크로태스크 실행 순서를 예측할 수 있다
- [ ] `requestAnimationFrame`의 실행 시점을 안다
- [ ] Node.js 이벤트 루프 단계를 안다

### 저장소·보안
- [ ] 각 저장소의 특성과 선택 기준을 안다
- [ ] XSS Stored/Reflected/DOM-based를 구분할 수 있다
- [ ] CSRF 방어법 3가지를 말할 수 있다
- [ ] CSP를 작성할 수 있다

### 캐싱
- [ ] Cache-Control 주요 값을 설명할 수 있다
- [ ] ETag 조건부 요청 흐름을 안다
- [ ] Service Worker 캐싱 전략 3가지를 안다
