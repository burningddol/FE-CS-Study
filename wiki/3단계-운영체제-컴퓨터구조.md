# 3단계: 운영체제·컴퓨터 구조

> 얕게 파되, FE 실무와 연결되는 지점은 반드시 짚어야 한다. JS가 싱글 스레드인 이유, 크롬이 멀티 프로세스인 이유, Node.js가 이벤트 기반인 이유를 OS 관점에서 설명할 수 있어야 한다.

**학습 기간**: 1주

* * *

## 1. 프로세스와 스레드

### 1-1. 프로세스
- 실행 중인 프로그램으로 정의한다. 자체 메모리 공간(코드·데이터·힙·스택)과 PCB를 갖는다
- 프로세스 간 통신은 반드시 IPC(파이프, 소켓, 공유 메모리)를 거쳐야 한다

### 1-2. 스레드
- 프로세스 내부의 실행 단위로 정의한다
- 같은 메모리 공간을 공유하므로 통신은 쉽지만 **동기화 문제**를 반드시 고려해야 한다
- 스택은 스레드별로 독립, 힙은 공유한다

### 1-3. 프로세스 vs 스레드
| | 프로세스 | 스레드 |
|---|---|---|
| 메모리 | 독립 | 공유 (힙, 데이터) |
| 생성 비용 | 큼 | 작음 |
| 통신 | IPC 필요 | 변수 공유 |
| 장애 격리 | O | X |

### 1-4. Context Switching
- CPU 레지스터, PC, 메모리 맵을 저장·복원한다
- 캐시 미스가 발생해 비용이 크다
- 스레드 간 전환이 프로세스 간 전환보다 싸다는 점을 기억한다

### 1-5. 크롬의 사이트 격리 아키텍처
- 탭마다 프로세스를 분리하고, 크로스 오리진 iframe도 별도 프로세스로 띄운다
- 탭 하나가 죽어도 전체가 죽지 않는 이유를 설명할 수 있어야 한다
- 메모리 사용량이 큰 이유도 같은 맥락에서 이해해야 한다

### 1-6. 웹 워커 계열
- **Web Worker**: 별도 스레드에서 실행한다. 메시지 패싱으로만 통신하고, DOM에는 접근할 수 없다
- **Service Worker**: 네트워크 프록시, 오프라인, 푸시를 담당한다
- **Shared Worker**: 여러 탭이 공유하는 워커로 활용한다
- **Worklet**: 렌더링 파이프라인의 특정 단계(Paint, Audio 등)에 삽입한다

### 1-7. 동시성 이슈
- Race Condition: 여러 스레드가 공유 자원에 동시 접근할 때 발생한다
- Deadlock: 서로의 자원을 기다리며 무한 대기한다
- 세마포어, 뮤텍스는 이름과 용도 정도는 알아야 한다
- **JS에서도 비동기 경쟁 상태는 발생한다.** API 응답 순서가 꼬이는 경우 `AbortController`, 요청 ID 체크로 방어해야 한다

## 2. 동기·비동기와 I/O 모델

### 2-1. 동기 vs 비동기, 블로킹 vs 논블로킹
네 가지 조합을 구분해야 한다. **호출자가 결과를 기다리는가**(동기)와 **호출자의 제어권이 즉시 반환되는가**(블로킹)는 별개의 축이다.

| | 블로킹 | 논블로킹 |
|---|---|---|
| 동기 | 일반 함수 호출 | 결과 폴링 |
| 비동기 | 잘 안 씀 | 콜백/Promise |

### 2-2. I/O 모델
**Node.js libuv가 왜 빠른가**의 맥락에서 이해해야 한다.

1. **Blocking I/O**: `read()` 호출 시 데이터가 올 때까지 스레드가 멈춘다 → 요청당 스레드가 필요해 확장성이 떨어진다
2. **Non-blocking I/O**: 즉시 반환되지만 폴링이 필요해 CPU를 낭비한다
3. **I/O Multiplexing** (`select`, `poll`, `epoll`, `kqueue`): 커널이 준비된 FD를 알려준다. **Node.js의 핵심**이다
4. **Asynchronous I/O**: 커널이 완료까지 처리한 뒤 통지한다

### 2-3. Node.js와 libuv
- 이벤트 루프는 **libuv**(C로 작성)가 제공한다
- 네트워크 I/O는 epoll/kqueue로 처리한다
- 파일 I/O는 내부 스레드 풀(기본 4개)을 사용한다
- 이것이 Node가 "싱글 스레드 기반인데도 빠른" 이유다

### 2-4. JS의 비동기 진화
- 콜백 → 콜백 지옥
- Promise → `then` 체인
- async/await → 동기처럼 보이는 비동기

## 3. 메모리 관리

### 3-1. 메모리 구조
- **Code (Text)**: 기계어
- **Data**: 전역 변수 (초기화된 영역 + BSS)
- **Heap**: 동적 할당, 아래 → 위로 증가
- **Stack**: 함수 호출 프레임, 위 → 아래로 증가

### 3-2. 스택 vs 힙
- 스택: 함수 지역 변수, LIFO, 빠름, 크기 제한(기본 ~1MB)
- 힙: `new`, 객체, 수명이 스코프와 무관

### 3-3. 가상 메모리
- 프로세스마다 독립된 주소 공간을 환상으로 제공한다
- 페이지 테이블로 물리 메모리에 매핑한다
- 페이지 폴트, 스와핑 개념을 안다

### 3-4. JS 가비지 컬렉션
- **Reachability** 기반으로 동작한다. 루트(전역, 현재 콜 스택)에서 도달 가능한 객체는 유지한다
- **Mark-and-Sweep**: 도달 가능한 것을 mark → 나머지를 sweep
- **세대별 GC** (V8):
  - Young Generation (Scavenger): 짧게 사는 객체, 빠르게 수거
  - Old Generation (Mark-Compact): 살아남은 객체, 느리지만 드물게 실행
- **증분 GC** (Incremental): 긴 멈춤을 피하기 위해 작게 쪼개 실행한다

### 3-5. 클로저와 메모리 누수
```js
function outer() {
  const big = new Array(1000000);
  return function inner() { /* big 사용 */ };
}
// inner가 살아있는 한 big도 GC되지 않는다
```

### 3-6. FE에서 자주 마주치는 메모리 누수 패턴
- 이벤트 리스너 미해제 (SPA 라우트 이동 시 특히 주의)
- setInterval 미정리
- detached DOM을 JS 변수가 계속 참조
- 전역 변수 오염

### 3-7. WeakMap / WeakSet / WeakRef
- key를 약한 참조로 두어 다른 참조가 없으면 GC를 허용한다
- 캐시, private 속성, 메타데이터 부착 등에 활용한다

### 3-8. DevTools 활용
- Memory 탭 Heap Snapshot으로 누수를 추적한다
- Performance 탭에서 GC 타이밍을 확인한다
- Allocation Timeline으로 할당 시점을 본다

## 4. CPU·캐시·지역성

### 4-1. CPU 기본 구조
- ALU(연산), 레지스터(초고속 저장), 제어 장치
- 파이프라이닝: 명령어 단계를 겹쳐 실행한다

### 4-2. 캐시 계층
| | 용량 | 속도 | 비용 |
|---|---|---|---|
| 레지스터 | 수백 바이트 | 1 cycle | 최고 |
| L1 | 수십 KB | ~4 cycles | |
| L2 | 수백 KB | ~12 cycles | |
| L3 | 수 MB | ~40 cycles | |
| RAM | GB | ~200 cycles | |
| SSD | TB | 수십만 cycles | 최저 |

### 4-3. 캐시 지역성
- **시간적 지역성**: 최근에 쓴 것을 다시 쓸 확률이 높다
- **공간적 지역성**: 근처 주소를 곧 쓸 확률이 높다
- 캐시는 보통 64바이트 단위(캐시 라인)로 로드한다

### 4-4. 배열 vs 연결 리스트의 실제 성능
- 이론적으로 연결 리스트 삽입은 O(1)이다
- 실제로는 배열이 대부분 더 빠르다 → 연속 메모리가 캐시에 친화적이기 때문이다
- "빅오가 같으면 상수가 작은 쪽이 이긴다"는 원칙을 기억한다

### 4-5. 분기 예측
- CPU는 다음 명령을 추측해서 파이프라이닝한다
- 정렬된 배열 순회가 정렬되지 않은 배열보다 빠른 이유가 여기에 있다
- 예측 실패 시 파이프라인 플러시가 일어나 느려진다

## 연습 문제

1. 크롬 탭을 하나 더 열 때 메모리가 급증하는 이유를 사이트 격리 관점에서 설명하라.
2. `fs.readFile`과 `fs.readFileSync`의 OS 수준 차이를 I/O 모델로 설명하라.
3. 아래 코드에서 메모리 누수가 발생하는 지점을 찾고 고쳐라.
   ```js
   function setup() {
     const data = new Array(100000);
     document.getElementById('btn').addEventListener('click', () => {
       console.log(data.length);
     });
   }
   ```
4. DevTools Performance 탭에서 Major GC 이벤트를 찾고, 직전에 무슨 일이 있었는지 분석하라.
5. `WeakMap`으로 객체에 private 속성을 구현하라.
6. 1천만 개 숫자의 합을 배열과 연결 리스트로 각각 구해 성능 차이를 측정하고, 캐시 라인으로 설명하라.
