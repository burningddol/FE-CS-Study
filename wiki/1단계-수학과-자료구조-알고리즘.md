# 1단계: 수학·논리 + 자료구조·알고리즘

> 전공자와 가장 많이 벌어지는 영역이다. **수학·논리 기초**를 먼저 잡고 자료구조/알고리즘으로 넘어간다. 단순 코테 풀이가 아니라, 알고리즘이 **왜** 그 복잡도를 갖는지 증명할 수 있는 수준이 목표다.

**학습 기간**: 2.5주

## 1. 수 표현과 비트 연산

컴퓨터가 숫자를 저장하는 실제 방식이다. **`0.1 + 0.2 !== 0.3`을 비트 수준에서 설명할 수 있어야 통과다.**

### 1-1. 2진수·16진수·2의 보수

#### 진법 변환
10진수 13을 2진수로 바꾸면 `1101`이다. 각 자리가 2의 거듭제곱을 의미한다.

```
13 = 8 + 4 + 0 + 1
   = 1*2³ + 1*2² + 0*2¹ + 1*2⁰
   = 1101₂
```

JS에서 진법 변환은 `toString(진법)` / `parseInt(문자열, 진법)`로 한다.

```js
(13).toString(2);       // "1101"
(255).toString(16);     // "ff"
parseInt("ff", 16);     // 255
parseInt("1101", 2);    // 13
```

#### 음수 표현: 2의 보수
양수 비트를 뒤집은 뒤(1의 보수) 1을 더하면 2의 보수가 된다. 왜 이렇게 하나? **덧셈 회로 하나로 뺄셈까지 처리**하기 위해서다.

```
 5를 8비트로:  0000 0101
 비트 반전:    1111 1010   (1의 보수)
 1 더함:       1111 1011   (2의 보수 = -5)

검증: 5 + (-5) = 0이어야 한다
  0000 0101
+ 1111 1011
-----------
1 0000 0000   ← 오버플로 비트 버리면 0
```

> **⚠️ 함정**: JS의 비트 연산은 **32비트 signed integer**로 내부 변환된다. `~5` 같은 연산이 `-6`인 이유가 여기 있다. (5의 32비트 2의 보수 반전 = -6)

```js
~5;   // -6
~0;   // -1
-1 >>> 0;  // 4294967295 (부호 없는 32비트로 해석)
```

### 1-2. IEEE 754 부동소수점

JS의 `Number`는 전부 **64비트 double**이다. 구성은 다음과 같다.

```
[부호 1비트][지수 11비트][가수 52비트]
```

값 = `(-1)^부호 × 1.가수 × 2^(지수 - 1023)` 형태로 저장한다. 편향값 1023을 빼는 이유는 지수 비트로 음수 지수도 표현하기 위해서다.

#### `0.1 + 0.2 !== 0.3`의 진짜 이유
`0.1`을 2진수로 바꾸면 **무한히 반복**한다.

```
0.1₁₀ = 0.0001100110011001100110011...₂   (0011 반복)
```

52비트에서 잘려 저장되므로 `0.1`도 `0.2`도 근사값이다. 둘을 더하면 오차가 누적되어 `0.30000000000000004`가 나온다.

```js
0.1 + 0.2;              // 0.30000000000000004
0.1 + 0.2 === 0.3;      // false
```

**올바른 비교 방법**: `Number.EPSILON`보다 작은 차이를 같다고 본다.

```js
Math.abs((0.1 + 0.2) - 0.3) < Number.EPSILON;  // true
```

#### 안전한 정수 범위
정수는 `-(2^53 - 1)` ~ `2^53 - 1`까지만 정확하다. 그 밖은 근사된다.

```js
Number.MAX_SAFE_INTEGER;       // 9007199254740991
Number.MAX_SAFE_INTEGER + 1;   // 9007199254740992
Number.MAX_SAFE_INTEGER + 2;   // 9007199254740992  ← 같아진다!
```

> **실무 규칙**: 서버에서 받은 `long` 타입 ID를 JS에서 처리할 때는 **문자열로 받아라**. 숫자로 받으면 정밀도 손실이 난다. 필요하면 `BigInt`를 쓴다.

```js
BigInt("9007199254740993") + 1n;  // 9007199254740994n
```

### 1-3. 비트 연산자

| 연산자 | 이름 | 예시 | 용도 |
|---|---|---|---|
| `&` | AND | `5 & 3 = 1` | 플래그 확인, 짝/홀 판정 |
| `\|` | OR | `5 \| 3 = 7` | 플래그 조합 |
| `^` | XOR | `5 ^ 3 = 6` | 토글, 두 값 swap |
| `~` | NOT | `~5 = -6` | 비트 반전 |
| `<<` | 왼쪽 시프트 | `1 << 3 = 8` | 2배 곱셈 |
| `>>` | 산술 시프트 | `-8 >> 1 = -4` | 부호 유지 나눗셈 |
| `>>>` | 논리 시프트 | `-1 >>> 0 = 4294967295` | 부호 비트도 0으로 |

#### 자주 쓰는 비트 트릭

```js
// 짝수/홀수 판정 (% 2보다 빠르다고는 못 함, JIT가 알아서 최적화)
const isEven = n => (n & 1) === 0;

// 2의 거듭제곱 판정
const isPowerOf2 = n => n > 0 && (n & (n - 1)) === 0;
isPowerOf2(8);   // true  (1000 & 0111 = 0)
isPowerOf2(6);   // false (0110 & 0101 = 0100)

// 특정 비트 켜기/끄기/토글
const setBit   = (n, i) => n |  (1 << i);
const clearBit = (n, i) => n & ~(1 << i);
const toggleBit = (n, i) => n ^ (1 << i);

// 가장 낮은 1 비트 추출
const lowestBit = n => n & -n;   // 12(1100) → 4(0100)
```

#### 플래그 집합 (비트마스크)

```js
const READ    = 1 << 0;  // 0b001 = 1
const WRITE   = 1 << 1;  // 0b010 = 2
const EXECUTE = 1 << 2;  // 0b100 = 4

const perm = READ | WRITE;            // 0b011 = 3
(perm & READ) !== 0;                  // true, 읽기 권한 있음
(perm & EXECUTE) !== 0;               // false
```

### 1-4. FE 실전 연결

- **권한 시스템 (RBAC)**: 사용자 권한을 비트마스크로 저장하면 배열 비교보다 빠르다
- **색상 계산**: `#FF8800` 같은 HEX를 `(r << 16) | (g << 8) | b`로 합성
- **Canvas / 이미지 픽셀**: 각 픽셀이 RGBA 32비트. 비트 마스킹으로 채널별 접근
- **성능 트릭**: `n << 1 === n * 2`, `n >> 1 === Math.floor(n / 2)` (단, 32비트 signed 정수에만 유효)

> **⚠️ 함정**: 비트 연산은 **부호 있는 32비트 정수로 강제 변환**한다. `2_500_000_000 << 1`은 오버플로로 음수가 된다. 일반 산술에선 `*`, `/`를 쓰고 비트 트릭은 꼭 필요할 때만 쓴다.

## 2. 이산수학 핵심

### 2-1. 명제 논리와 드모르간 법칙

**드모르간 법칙**은 복잡한 조건문을 단순화할 때 쓴다.

```
!(A && B) ≡ (!A || !B)
!(A || B) ≡ (!A && !B)
```

#### 실전 리팩토링

```js
// Before
if (!(user && user.isActive)) {
  return null;
}

// After (드모르간 + 가독성)
if (!user || !user.isActive) {
  return null;
}

// 또는 옵셔널 체이닝
if (!user?.isActive) {
  return null;
}
```

> **실무 규칙**: 조건문이 `!(A && B)` 형태로 보이면 바로 드모르간으로 풀어낸다. 리뷰어가 한눈에 읽을 수 있다.

### 2-2. 집합과 배열 메서드

수학의 집합 연산은 JS에서 다음과 같이 대응된다.

| 집합 연산 | 기호 | JS 구현 |
|---|---|---|
| 합집합 | A ∪ B | `[...new Set([...a, ...b])]` |
| 교집합 | A ∩ B | `a.filter(x => bSet.has(x))` |
| 차집합 | A - B | `a.filter(x => !bSet.has(x))` |
| 부분집합 | A ⊆ B | `a.every(x => bSet.has(x))` |

```js
const a = [1, 2, 3, 4];
const b = [3, 4, 5, 6];
const bSet = new Set(b);

const union =        [...new Set([...a, ...b])];       // [1,2,3,4,5,6]
const intersection = a.filter(x => bSet.has(x));       // [3,4]
const difference =   a.filter(x => !bSet.has(x));      // [1,2]
```

> **⚠️ 함정**: `a.filter(x => b.includes(x))`는 O(n·m)이다. `b`를 Set으로 감싸면 O(n + m)이 된다. 큰 배열에선 수십 배 차이가 난다.

### 2-3. 순열·조합

- **순열** (`nPr`): 순서 있음, `n! / (n-r)!`
- **조합** (`nCr`): 순서 없음, `n! / (r! * (n-r)!)`
- **중복 순열**: 같은 원소 여러 번, `n^r`

백트래킹의 이론적 기반이다.

```js
// 조합 생성 (nCr)
function combinations(arr, r) {
  if (r === 0) return [[]];
  if (arr.length === 0) return [];
  const [first, ...rest] = arr;
  const withFirst = combinations(rest, r - 1).map(c => [first, ...c]);
  const withoutFirst = combinations(rest, r);
  return [...withFirst, ...withoutFirst];
}

combinations([1, 2, 3, 4], 2);
// [[1,2],[1,3],[1,4],[2,3],[2,4],[3,4]]  →  4C2 = 6개
```

### 2-4. 모듈러 연산

`a mod n`은 `a`를 `n`으로 나눈 나머지다.

- 해시 테이블이 `hash(key) % bucketCount`를 쓰는 이유는 결과값을 인덱스 범위에 맞추기 위해서다
- 원형 버퍼에서 인덱스를 순환시킬 때 쓴다: `(i + 1) % size`

> **⚠️ 함정**: 음수에 대한 `%` 결과는 언어마다 다르다. JS는 **피제수의 부호**를 따른다.

```js
-1 % 5;          // -1  (JS/C/Java)
// 파이썬은 4를 반환한다

// 항상 양수로 받고 싶다면
const mod = (a, n) => ((a % n) + n) % n;
mod(-1, 5);      // 4
```

### 2-5. 수학적 귀납법

재귀의 정당성과 반복문 불변량(invariant) 증명의 기초다.

- **기저 단계**: 가장 작은 경우에 참
- **귀납 단계**: n일 때 참이면 n+1일 때도 참

```js
// 팩토리얼의 귀납적 정의
// 기저: factorial(0) = 1
// 귀납: factorial(n) = n * factorial(n - 1)
function factorial(n) {
  if (n === 0) return 1;
  return n * factorial(n - 1);
}
```

루프로 쓸 때의 불변량 사고:

```js
// 불변량: 매 루프 종료 시점에 result === nums[0..i]의 합
function sum(nums) {
  let result = 0;
  for (let i = 0; i < nums.length; i++) {
    result += nums[i];   // 불변량 유지
  }
  return result;
}
```

## 3. 복잡도 분석

### 3-1. 빅오·빅오메가·빅세타

수학적 정의가 중요하다. "어떤 상수 c와 n₀가 존재해서..."로 시작한다.

| 표기 | 의미 | 직관 |
|---|---|---|
| O(f) | 상한 (upper bound) | "최악에도 이보다 나쁠 순 없다" |
| Ω(f) | 하한 (lower bound) | "최선에도 이보다 좋을 순 없다" |
| Θ(f) | 정확한 차수 | 상·하한이 일치 |

면접에서는 "빅오"만 쓰는 경우가 많지만, 퀵 정렬처럼 **평균 O(n log n), 최악 O(n²)** 인 경우 차이를 말할 수 있어야 한다.

### 3-2. 대표 복잡도 직관

| 복잡도 | 예시 | n=1,000,000 대략 시간 |
|---|---|---|
| O(1) | 해시 조회, 배열 인덱싱 | 즉시 |
| O(log n) | 이분 탐색, B-Tree | 20 step |
| O(n) | 선형 탐색, 단순 순회 | 백만 step |
| O(n log n) | 효율 정렬의 하한 | 2천만 step |
| O(n²) | 이중 루프, 버블 정렬 | 1조 step — 불가능 |
| O(2ⁿ) | 부분집합, 나이브 피보나치 | n=30만 되어도 폭발 |

> **실무 규칙**: n이 10⁴까지면 O(n²)도 괜찮다. 10⁵부터는 O(n log n)이 필요하다. 10⁶ 이상은 O(n) 또는 O(log n)이 필요하다.

### 3-3. 분할상환 분석 (Amortized)

단일 연산은 느려도 **일련의 연산 전체 평균**이 빠르면 "분할상환 O(1)"이라 부른다.

#### 동적 배열의 push
배열이 꽉 차면 2배 크기의 새 배열을 할당하고 복사한다(O(n)). 하지만 이후 n번의 push는 복사 없이 O(1)이다.

```
size 1 → 2 (복사 1)
size 2 → 4 (복사 2)
size 4 → 8 (복사 4)
...
n개 push의 총 비용 = 1 + 2 + 4 + ... ≈ 2n = O(n)
→ 평균 O(1)
```

### 3-4. JS 내부의 숨은 복잡도

| 메서드 | 복잡도 | 비고 |
|---|---|---|
| `Array.sort()` | O(n log n) | V8은 Timsort |
| `Array.indexOf()` / `includes()` | O(n) | 큰 배열은 Set으로 교체 |
| `Array.push()` / `pop()` | 분할상환 O(1) | 끝에서 작업 |
| `Array.unshift()` / `shift()` | O(n) | 전부 재배치 |
| `String.includes()` | 단순 O(n·m) | KMP 아님 |
| `Object` 키 조회 | O(1) 평균 | 해시 기반 |
| `Map.get()` / `Set.has()` | O(1) 평균 | |
| `JSON.stringify()` | O(n) | 깊이 주의 (stack overflow) |

## 4. 배열·문자열·해시

### 4-1. 배열 연산 비용

연속 메모리라서 **인덱스 접근은 O(1)**이지만, **앞쪽에서의 삽입/삭제는 O(n)**이다. 뒤에서 작업해야 한다.

```js
const arr = [1, 2, 3, 4, 5];

// O(1) — 끝에서 작업
arr.push(6);       // [1,2,3,4,5,6]
arr.pop();         // [1,2,3,4,5]

// O(n) — 앞에서 작업, 모든 요소 한 칸씩 이동
arr.unshift(0);    // [0,1,2,3,4,5]
arr.shift();       // [1,2,3,4,5]
```

> **실무 규칙**: 큐를 구현할 때 배열의 `shift()`를 쓰면 O(n)이다. 큐가 큰 경우 head 인덱스 트릭이나 연결 리스트 기반 큐를 고려한다.

### 4-2. 문자열 불변성

JS 문자열은 **immutable**이다. `+=`로 연결할 때마다 새 문자열 객체가 생긴다.

```js
// ❌ O(n²) — n번 반복하면 1+2+3+...+n = O(n²)
let s = "";
for (let i = 0; i < 10000; i++) {
  s += i.toString();
}

// ✅ O(n) — 배열에 모았다가 한 번에 합침
const parts = [];
for (let i = 0; i < 10000; i++) {
  parts.push(i.toString());
}
const result = parts.join("");
```

> **참고**: V8은 `+=` 연쇄를 일부 최적화하지만(Cons String), 패턴에 의존하지 말고 `join`을 쓰는 게 안전하다.

### 4-3. 투 포인터 / 슬라이딩 윈도우

정렬된 배열이나 문자열에서 O(n²)를 O(n)으로 줄이는 패턴이다.

#### 정렬된 배열에서 두 수의 합
```js
function twoSum(nums, target) {
  let left = 0, right = nums.length - 1;
  while (left < right) {
    const sum = nums[left] + nums[right];
    if (sum === target) return [left, right];
    if (sum < target) left++;
    else right--;
  }
  return null;
}
twoSum([1, 3, 5, 7, 9], 12);  // [2, 4]
```

#### 슬라이딩 윈도우 — 연속 부분합 최댓값
```js
function maxSumSubarray(nums, k) {
  let windowSum = 0;
  for (let i = 0; i < k; i++) windowSum += nums[i];
  let maxSum = windowSum;
  for (let i = k; i < nums.length; i++) {
    windowSum += nums[i] - nums[i - k];  // 한 칸 슬라이드
    maxSum = Math.max(maxSum, windowSum);
  }
  return maxSum;
}
```

### 4-4. 해시 테이블 내부

해시 함수가 키를 버킷 인덱스로 바꾼다. 버킷이 배열이고 인덱스 접근이 O(1)이기 때문에 조회가 O(1)이다.

```
key "apple"  → hash() → 42 → bucket[42 % 16] = bucket[10]
key "banana" → hash() → 58 → bucket[58 % 16] = bucket[10]  ← 충돌!
```

#### 충돌 해결
- **체이닝**: 버킷에 연결 리스트를 달고 같은 버킷의 키를 나열 (JS Map에 가깝다)
- **개방 주소법**: 다음 빈 버킷을 찾아 저장 (Python dict)

#### 평균 O(1), 최악 O(n)
해시 함수가 나쁘거나 공격자가 특정 키를 몰아넣으면 모두 같은 버킷에 쌓인다. 이것이 **HashDoS** 공격의 원리다.

### 4-5. Object vs Map vs Set

| | Object | Map | Set |
|---|---|---|---|
| 키 타입 | 문자열/심볼만 | 모든 값 | N/A (값) |
| 삽입 순서 | ES2015+ 보장 | 보장 | 보장 |
| 크기 조회 | `Object.keys().length` O(n) | `.size` O(1) | `.size` O(1) |
| 프로토타입 | 오염 가능 (`__proto__` 등) | 깨끗 | 깨끗 |
| 순회 | `Object.entries()` | 직접 `for-of` | 직접 `for-of` |
| JSON 변환 | 쉬움 | 직접 변환 필요 | 직접 변환 필요 |

```js
// Object는 프로토타입 오염 위험
const obj = {};
obj["__proto__"] = { polluted: true };   // 위험

// Map은 안전

map.set("__proto__", { polluted: true });  // 그냥 키다
```

> **실무 규칙**: 외부 입력을 키로 받는 딕셔너리는 반드시 `Map` 또는 `Object.create(null)`을 쓴다.

## 5. 스택·큐·힙

### 5-1. 스택 (LIFO)

**Last In, First Out**. 나중에 들어간 게 먼저 나간다.

#### 실전 예시
- 함수 **콜 스택**
- 브라우저 **뒤로가기 히스토리**
- **Undo / Redo**
- **괄호 검증**

```js
function isValidParentheses(s) {
  const stack = [];
  const pairs = { ")": "(", "]": "[", "}": "{" };
  for (const ch of s) {
    if ("([{".includes(ch)) stack.push(ch);
    else if (stack.pop() !== pairs[ch]) return false;
  }
  return stack.length === 0;
}
isValidParentheses("(a[b{c}]d)");   // true
isValidParentheses("(a[b}c)");       // false
```

### 5-2. 큐 (FIFO)

**First In, First Out**. 순서대로 처리한다.

- 이벤트 루프의 **태스크 큐 / 마이크로태스크 큐** (2단계에서 심화)
- **BFS** 구현
- **스로틀링**된 API 호출 대기열

```js
// 배열 기반 큐 (작은 규모만)
const queue = [];
queue.push(1);     // enqueue
queue.push(2);
queue.shift();     // dequeue → 1, but O(n)!

// 큰 규모는 head 인덱스 트릭
class Queue {
  #items = [];
  #head = 0;
  enqueue(v) { this.#items.push(v); }
  dequeue() {
    if (this.#head >= this.#items.length) return undefined;
    const v = this.#items[this.#head++];
    if (this.#head > 1000) { this.#items = this.#items.slice(this.#head); this.#head = 0; }
    return v;
  }
  get size() { return this.#items.length - this.#head; }
}
```

### 5-3. 덱 (Deque)

양쪽 끝에서 삽입/삭제가 O(1). 슬라이딩 윈도우 최댓값 같은 문제에서 핵심.

### 5-4. 힙과 우선순위 큐 (★ 직접 구현)

**우선순위 큐**는 "가장 작은(또는 큰) 것"을 빠르게 꺼내야 하는 경우의 자료구조다. 배열로 구현하면 삽입 O(n) 또는 추출 O(n) 중 하나가 느리다. 힙은 **둘 다 O(log n)** 이다.

#### 이진 힙의 구조
완전 이진 트리 + 힙 속성(부모 ≤ 자식, MinHeap 기준).

배열로 표현할 때 인덱스 관계:
- 부모: `(i - 1) / 2`
- 왼쪽 자식: `2*i + 1`
- 오른쪽 자식: `2*i + 2`

```
     1            배열: [1, 3, 2, 6, 5, 4]
    / \           인덱스: 0  1  2  3  4  5
   3   2
  / \ /
 6  5 4
```

#### MinHeap 구현

```js
class MinHeap {
  #heap = [];

  push(v) {
    this.#heap.push(v);
    this.#siftUp(this.#heap.length - 1);
  }

  pop() {
    if (this.#heap.length === 0) return undefined;
    const top = this.#heap[0];
    const last = this.#heap.pop();
    if (this.#heap.length > 0) {
      this.#heap[0] = last;
      this.#siftDown(0);
    }
    return top;
  }

  peek() { return this.#heap[0]; }
  get size() { return this.#heap.length; }

  #siftUp(i) {
    while (i > 0) {
      const parent = (i - 1) >> 1;
      if (this.#heap[parent] <= this.#heap[i]) break;
      [this.#heap[parent], this.#heap[i]] = [this.#heap[i], this.#heap[parent]];
      i = parent;
    }
  }

  #siftDown(i) {
    const n = this.#heap.length;
    while (true) {
      const l = 2 * i + 1, r = 2 * i + 2;
      let smallest = i;
      if (l < n && this.#heap[l] < this.#heap[smallest]) smallest = l;
      if (r < n && this.#heap[r] < this.#heap[smallest]) smallest = r;
      if (smallest === i) break;
      [this.#heap[smallest], this.#heap[i]] = [this.#heap[i], this.#heap[smallest]];
      i = smallest;
    }
  }
}

const h = new MinHeap();
[5, 1, 4, 2, 3].forEach(v => h.push(v));
h.pop(); h.pop();  // 1, 2
```

**다익스트라, A\*, Top-K 문제**의 핵심 자료구조다.

## 6. 연결 리스트·트리·그래프

### 6-1. 연결 리스트

노드가 다음 노드를 가리키는 구조. 배열과 달리 연속 메모리가 아니다.

| | 배열 | 연결 리스트 |
|---|---|---|
| 임의 접근 | O(1) | O(n) |
| 앞 삽입/삭제 | O(n) | O(1) |
| 메모리 | 연속 | 분산 (포인터 오버헤드) |
| 캐시 효율 | 좋음 | 나쁨 |

> **실무 팁**: 이론상 삽입 O(1)이지만 **실제 성능에선 배열이 이기는 경우가 많다**. 캐시 지역성 때문이다 (3단계에서 심화).

#### 연결 리스트 뒤집기 (포인터 3개 패턴)

```js
function reverseList(head) {
  let prev = null, curr = head;
  while (curr) {
    const next = curr.next;   // 다음 노드 저장
    curr.next = prev;         // 방향 전환
    prev = curr;              // 전진
    curr = next;
  }
  return prev;                // 새 head
}
```

### 6-2. 이진 트리·BST

#### 순회 4가지
- **전위 (preorder)**: 루트 → 왼쪽 → 오른쪽. 트리 복사, 표현식 트리
- **중위 (inorder)**: 왼쪽 → 루트 → 오른쪽. **BST에서 정렬된 순서로 방문**
- **후위 (postorder)**: 왼쪽 → 오른쪽 → 루트. 트리 삭제, 후위 표기식
- **레벨 (BFS)**: 층별로. 큐 사용

```js
function inorder(node, result = []) {
  if (!node) return result;
  inorder(node.left, result);
  result.push(node.value);
  inorder(node.right, result);
  return result;
}
```

#### BST (이진 탐색 트리)
- 왼쪽 < 루트 < 오른쪽 불변량
- 탐색/삽입/삭제: 균형일 때 O(log n), 편향 시 최악 O(n)
- **AVL, Red-Black 트리**는 회전으로 자동 균형을 유지한다 (이름만 기억)

### 6-3. 트라이 (Trie)

문자열 특화 트리. 자동완성의 핵심 자료구조.

```
insert: "car", "cat", "cup"

        (root)
        /    \
       c      ...
      / \
     a   u
    /|   \
   r t    p
   *  *    *   (* = 단어 끝 표시)
```

- 삽입/검색: O(m), m = 문자열 길이 (단어 수와 무관)
- 공간: O(문자 총 수)
- **실전**: 주소 자동완성, 태그 필터링, IDE 자동완성

```js
class Trie {
  #root = {};

  insert(word) {
    let node = this.#root;
    for (const ch of word) {
      if (!node[ch]) node[ch] = {};
      node = node[ch];
    }
    node.$ = true;  // 단어 끝
  }

  startsWith(prefix) {
    let node = this.#root;
    for (const ch of prefix) {
      if (!node[ch]) return false;
      node = node[ch];
    }
    return true;
  }
}
```

### 6-4. Union-Find (Disjoint Set)

"이 두 노드가 같은 집합에 속하는가?"를 빠르게 판정한다.

- **Union**: 두 집합 병합
- **Find**: 어떤 집합에 속하는지

두 최적화를 함께 쓰면 **거의 O(1)** 이다:
- **Path Compression**: find 시 경로 위 모든 노드를 루트에 직접 연결
- **Union by Rank**: 낮은 트리를 높은 트리 밑에 붙임

```js
class UnionFind {
  #parent; #rank;
  constructor(n) {
    this.#parent = Array.from({ length: n }, (_, i) => i);
    this.#rank = new Array(n).fill(0);
  }
  find(x) {
    if (this.#parent[x] !== x) this.#parent[x] = this.find(this.#parent[x]);
    return this.#parent[x];
  }
  union(a, b) {
    const ra = this.find(a), rb = this.find(b);
    if (ra === rb) return false;
    if (this.#rank[ra] < this.#rank[rb]) this.#parent[ra] = rb;
    else if (this.#rank[ra] > this.#rank[rb]) this.#parent[rb] = ra;
    else { this.#parent[rb] = ra; this.#rank[ra]++; }
    return true;
  }
}
```

**MST (Kruskal)**, 친구 관계 탐색, 네트워크 연결성 판정에 쓴다.

### 6-6. DFS / BFS

#### DFS (깊이 우선)
```js
function dfs(graph, start) {
  const visited = new Set();
  function go(node) {
    if (visited.has(node)) return;
    visited.add(node);
    for (const next of graph[node]) go(next);
  }
  go(start);
  return visited;
}
```

#### BFS (너비 우선) — 최단 경로(가중치 없음)
```js
function bfs(graph, start) {
  const visited = new Set([start]);
  const queue = [start];
  const order = [];
  while (queue.length) {
    const node = queue.shift();
    order.push(node);
    for (const next of graph[node]) {
      if (!visited.has(next)) {
        visited.add(next);
        queue.push(next);
      }
    }
  }
  return order;
}
```

> **FE 연결**: **DOM 트리 순회**, **Virtual DOM diff**, **컴포넌트 트리 리렌더**, **번들러 의존성 그래프**(위상 정렬)가 전부 그래프 탐색이다.

### 7-2. 이분 탐색

정렬된 배열에서 O(log n).

```js
function binarySearch(arr, target) {
  let lo = 0, hi = arr.length - 1;
  while (lo <= hi) {
    const mid = (lo + hi) >> 1;  // (lo + hi) / 2, 안전하게 비트 시프트
    if (arr[mid] === target) return mid;
    if (arr[mid] < target) lo = mid + 1;
    else hi = mid - 1;
  }
  return -1;
}
```

#### lower_bound / upper_bound
- `lower_bound`: target **이상**의 첫 위치
- `upper_bound`: target **초과**의 첫 위치

#### 파라메트릭 서치
"최대 속도가 얼마?" 같은 최적화 문제를 "속도 X가 가능한가?"라는 결정 문제로 바꿔서 이분 탐색한다.

### 7-3. 재귀와 반복

재귀는 **기저 + 귀납**으로 정의한다.

> **⚠️ 함정**: JS 엔진 대부분은 **꼬리 재귀 최적화(TCO)를 지원하지 않는다**. 깊은 재귀는 stack overflow가 난다. Safari만 부분적으로 지원한다.

```js
// 10만 번 재귀 — 대부분 브라우저에서 RangeError

}

```

해결책은 반복문으로 전환, 또는 명시적 스택 사용.

### 7-4. 백트래킹

모든 경우를 시도하되, **가능성 없는 가지는 잘라낸다(pruning)**.

```js
// N-Queens
function solveNQueens(n) {
  const result = [];
  const board = Array(n).fill(-1);  // board[row] = col
  function backtrack(row) {
    if (row === n) { result.push([...board]); return; }
    for (let col = 0; col < n; col++) {
      let ok = true;
      for (let r = 0; r < row; r++) {
        if (board[r] === col || Math.abs(board[r] - col) === row - r) {
          ok = false; break;
        }
      }
      if (ok) { board[row] = col; backtrack(row + 1); }
    }
  }
  backtrack(0);
  return result;
}
```

### 7-5. 분할 정복

문제를 반으로 쪼개 각각 풀고 합친다. 머지 정렬, 퀵 정렬이 대표.

### 7-6. 동적 계획법 (DP)

**최적 부분 구조** + **중복 부분 문제**가 있을 때 쓴다.

#### 메모이제이션 (Top-down) vs 타뷸레이션 (Bottom-up)

```js
// Top-down — 재귀 + 캐시
function fib(n, memo = {}) {
  if (n < 2) return n;
  if (memo[n] !== undefined) return memo[n];
  return memo[n] = fib(n - 1, memo) + fib(n - 2, memo);
}

// Bottom-up — 반복 + 배열
function fibDP(n) {
  if (n < 2) return n;
  const dp = [0, 1];
  for (let i = 2; i <= n; i++) dp[i] = dp[i - 1] + dp[i - 2];
  return dp[n];
}
```

#### 대표 DP 문제
- **피보나치**: `dp[i] = dp[i-1] + dp[i-2]`
- **LIS (최장 증가 수열)**: `dp[i] = max(dp[j]) + 1 for j < i, arr[j] < arr[i]`
- **LCS (최장 공통 부분 수열)**: 2차원 DP
- **배낭 문제**: 2차원 DP
- **동전 교환**: 1차원 DP

### 7-7. 그리디

매 단계에서 "지금 최선"을 고른다. **정당성 증명**이 핵심이다.

> **⚠️ 함정**: "그리디로 될 것 같은데 안 되는 문제"가 많다. 반례를 들 수 없어야 그리디를 쓴다. 일반적으론 DP로 풀고, 특수 조건에서만 그리디가 최적이다.

### 7-8. 위상 정렬

DAG(방향성 비순환 그래프)에서 선후 관계를 맞춰 나열한다. **번들러 의존성 해결, 태스크 스케줄링**의 원리.

#### Kahn 알고리즘 (진입 차수 기반)
```js
function topologicalSort(graph) {
  const indegree = {};
  for (const node in graph) indegree[node] = 0;
  for (const node in graph) for (const next of graph[node]) indegree[next] = (indegree[next] || 0) + 1;

  const queue = Object.keys(indegree).filter(n => indegree[n] === 0);
  const result = [];
  while (queue.length) {
    const node = queue.shift();
    result.push(node);
    for (const next of graph[node] || []) {
      indegree[next]--;
      if (indegree[next] === 0) queue.push(next);
    }
  }
  if (result.length !== Object.keys(indegree).length) throw new Error("cycle detected");
  return result;
}
```

### 7-9. 최단 경로

| 알고리즘 | 복잡도 | 특징 |
|---|---|---|
| BFS | O(V + E) | **가중치 없는** 최단 경로 |
| 다익스트라 | O((V + E) log V) | 음수 가중치 불가, 우선순위 큐 |
| 벨만-포드 | O(VE) | 음수 가중치 허용, 음수 사이클 탐지 |
| 플로이드-워셜 | O(V³) | 모든 쌍 사이 최단 경로 |

### 7-10. 편집 거리 (Levenshtein)

두 문자열을 같게 만드는 최소 편집(삽입/삭제/교체) 횟수.

```js
function editDistance(a, b) {
  const m = a.length, n = b.length;
  const dp = Array.from({ length: m + 1 }, () => new Array(n + 1).fill(0));
  for (let i = 0; i <= m; i++) dp[i][0] = i;
  for (let j = 0; j <= n; j++) dp[0][j] = j;
  for (let i = 1; i <= m; i++) {
    for (let j = 1; j <= n; j++) {
      if (a[i - 1] === b[j - 1]) dp[i][j] = dp[i - 1][j - 1];
      else dp[i][j] = 1 + Math.min(dp[i - 1][j], dp[i][j - 1], dp[i - 1][j - 1]);
    }
  }
  return dp[m][n];
}
editDistance("kitten", "sitting");  // 3
```

**git diff, 검색 유사도, 오타 제안**이 전부 편집 거리 기반이다. Virtual DOM의 diff도 **트리 편집 거리의 단순화 버전**이다 (O(n³)을 O(n)으로 만들기 위한 휴리스틱).

2. 정수 n이 2의 거듭제곱인지 판정하는 함수를 비트 연산으로 작성하라.
3. 배열에서 중복 제거를 Object / Map / Set 세 방법으로 구현하고 복잡도를 비교하라.

### 심화
4. MinHeap을 배열로 직접 구현하고, 정수 배열에서 Top-K 최댓값을 O(n log k)로 구하라.
5. 문자열 배열을 트라이에 넣고 접두사 검색 함수를 작성하라.
6. 모듈 의존성이 `{A: [B, C], B: [D], C: [D], D: []}`일 때 위상 정렬을 구현하라.
7. `"kitten"`과 `"sitting"`의 편집 거리를 DP 테이블로 그려라.

### FE 연결
8. 디바운스 함수를 구현하고, 그 동작이 왜 "태스크 큐"를 건드리는지 설명하라.
9. Virtual DOM diff가 왜 편집 거리의 단순화 버전인지 설명하라.

## 연습 문제 정답 / 힌트

**1.**
```js
(0.1 + 0.2).toFixed(20);  // "0.30000000000000004441"
```
`0.1`과 `0.2`는 2진수로 무한 반복 소수다. 64비트 double의 52비트 가수에서 반올림되어 저장된 근사값을 더해 오차가 드러난 것.

**2.**
```js
const isPowerOf2 = n => n > 0 && (n & (n - 1)) === 0;
```
2의 거듭제곱은 단 하나의 1비트를 가진다. `n - 1`을 하면 그 비트 밑으로 1이 채워져 겹치지 않는다.

**3.**
```js
// Set — 가장 간결, O(n)
[...new Set(arr)];

// Map — 순서 유지 + 추가 정보
const m = new Map();
for (const v of arr) if (!m.has(v)) m.set(v, true);
[...m.keys()];

// Object — 문자열 키로 바뀌므로 타입 섞이면 위험
const seen = {};
for (const v of arr) seen[v] = true;
Object.keys(seen);   // 전부 문자열이 됨!
```
Object는 숫자 키를 문자열로 바꿔 저장한다. `1`과 `"1"`을 같다고 본다. Map/Set이 안전하다.

**6.**
Kahn 알고리즘으로 진입 차수 0인 노드부터 차례로 빼면 `[A, B, C, D]` 또는 `[A, C, B, D]`가 나온다. 둘 다 유효한 위상 정렬 결과다.

**9.**
Virtual DOM diff는 "트리 A를 트리 B로 바꾸는 최소 연산"을 계산하는 문제다. 일반 트리 편집 거리는 O(n³)이지만, React는 두 가지 휴리스틱으로 O(n)에 맞춘다:
1. **다른 타입의 노드는 하위 트리 전체를 새로 만든다** (교차 비교 안 함)
2. **같은 레벨의 리스트는 `key`로 매칭한다**

이 휴리스틱 때문에 `key` prop이 중요하고, 리스트 순서가 바뀌는데 key가 인덱스면 성능이 망가진다.

## 체크리스트

- [ ] 부동소수점 오차의 원인을 비트 수준에서 설명할 수 있다
- [ ] `& | ^ << >> >>>` 각각의 용도를 즉답할 수 있다
- [ ] 드모르간 법칙으로 복잡한 `if` 조건을 리팩토링할 수 있다
- [ ] 빅오를 상한의 수학적 정의로 말할 수 있다
- [ ] `Array.sort()`가 내부에서 무엇을 쓰는지 안다
- [ ] 해시 테이블이 최악 O(n)인 조건을 안다
- [ ] MinHeap을 배열로 구현할 수 있다
- [ ] BFS와 DFS의 구현 차이를 스택/큐로 설명할 수 있다
- [ ] 트라이의 시간/공간 복잡도를 안다
- [ ] Union-Find의 최적화 기법 2개를 말할 수 있다
- [ ] 위상 정렬을 Kahn 알고리즘으로 구현할 수 있다
- [ ] 다익스트라가 왜 우선순위 큐를 쓰는지 설명할 수 있다
- [ ] DP와 그리디의 판단 기준을 안다
- [ ] Timsort가 Merge/Insertion의 하이브리드인 이유를 안다
- [ ] 편집 거리 DP를 칠판에 그릴 수 있다
