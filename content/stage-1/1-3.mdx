## 왜 이 장인가

"이 코드가 느린가?"를 답하는 방식에는 두 가지가 있다.

하나는 **측정**이다. 실제로 돌려보고 시간을 잰다. 다른 하나는 **예측**이다. 입력이 커졌을 때 얼마나 나빠질지를 수학적으로 가늠한다. 측정은 구체적이지만, 측정 환경이 달라지면 무너진다. 예측은 환경과 무관하게 **입력 크기 대비 증가율**만 본다.

복잡도 분석은 이 두 번째 방식의 언어다. `O(n)`, `O(n log n)`, `O(n²)`는 "얼마나 느린가"가 아니라 **"얼마나 빨리 나빠지는가"**를 말한다.

## § 1 · 성장률이 전부다

<ComplexityChart />

입력이 10배로 늘어날 때 연산량이:

- 같으면 `O(1)`
- 3~4배면 `O(log n)`
- 10배면 `O(n)`
- 30~40배면 `O(n log n)`
- 100배면 `O(n²)`
- 10억 배 이상이면 `O(2ⁿ)`

`n = 1,000,000`일 때 각 복잡도의 연산 수를 비교해 보자.

| 복잡도 | 대략 연산 수 | 체감 |
|---|---|---|
| O(log n) | 20 | 즉시 |
| O(n) | 1,000,000 | 즉시 |
| O(n log n) | 20,000,000 | 수십 ms |
| O(n²) | 10¹² | **분 단위** |
| O(2ⁿ) | 우주가 끝나기 전까지 | 불가능 |

그래서 "O(n²)는 O(n)보다 느리다"가 아니라 **"입력이 커지면 O(n²)는 사실상 멈춘다"**가 정확한 감각이다.

## § 2 · 빅오·빅오메가·빅세타의 구분

- **Big O** — `O(f(n))`: 상한 — 최악의 경우 이보다 나쁘진 않다
- **Big Ω** — `Ω(f(n))`: 하한 — 최선의 경우 이보다 좋진 않다
- **Big Θ** — `Θ(f(n))`: 상한과 하한이 같을 때

실무에선 대부분 Big O만 쓴다. 하지만 "최선의 경우와 최악의 경우가 어떻게 다른가"를 말할 일은 반드시 생긴다(퀵 정렬이 대표적이다).

```js
// Array.indexOf
// 최선 Ω(1)  — 첫 요소가 찾는 값
// 최악 O(n) — 마지막 요소거나 없음
// 평균 Θ(n)
```

## § 3 · 분할상환 분석 — 평균을 말하는 법

`Array.push`는 왜 "평균 O(1)"이라고 할까?

내부적으로 JS 배열은 **고정 용량**을 갖는다. 용량이 꽉 차면 **2배로 늘려** 새 배열에 복사한다. 이 복사가 O(n). 나머지 push는 O(1).

`n`번 push할 때 총 복사량은 `1 + 2 + 4 + … + n ≈ 2n`. 즉 `n`번 push에 `O(n)` 작업. **한 번당 평균 O(1)**이 된다.

> 이것이 "분할상환(amortized) O(1)"이다. 개별 연산이 최악 O(n)이어도, **여러 번 모아서 평균 내면** O(1)이 된다.

## § 4 · JS 내부에 숨은 복잡도

같은 연산처럼 보여도 내부 구현은 다르다.

```js
arr.sort();               // Timsort, 평균 O(n log n)
arr.indexOf(x);           // 선형 스캔, O(n)
set.has(x);               // 해시, 평균 O(1)
'abc'.includes('b');      // 단순 문자열 검색, O(n·m) — KMP 아님
str.charAt(i);            // O(1)
arr.splice(0, 1);         // O(n)  ← 조심
arr.push(x);              // 평균 O(1)
arr.unshift(x);           // O(n)  ← 전부 오른쪽으로 밀기
```

**`indexOf` + 반복문**은 언제나 의심해야 한다. `n`개의 배열에서 각 원소를 찾는 `indexOf` 반복이 있다면 **O(n²)**. 그 자리를 `Set.has`로 바꾸면 **O(n)**.

## § 5 · 상수는 버리지만, 무시하진 않는다

이론적으로 `O(2n) = O(n)`이다. 하지만 현실에서 2배 차이는 2배 차이다. 특히 **캐시 적중률**이 다르면 같은 O(n)이어도 10배 이상 벌어질 수 있다.

그래서 실무의 감각은 이렇게 나뉜다.

- **설계 단계**: Big O로 말한다 — "이건 O(n²)니까 안 됨"
- **최적화 단계**: 프로파일링으로 말한다 — "이게 총 시간의 40%를 차지함"

## § 6 · 체크포인트

- [ ] `O(n²)`가 왜 `n`이 커질수록 "느려진다"가 아니라 "멈춘다"가 되는지 감이 잡힌다
- [ ] `Array.push`의 분할상환 O(1)을 2배 확장 전략으로 설명할 수 있다
- [ ] `Array.indexOf`를 반복문 안에 쓰는 것이 왜 O(n²)인지 안다
- [ ] Big O와 Big Θ의 차이를 `Array.indexOf`로 예시를 들 수 있다
- [ ] 이론적 복잡도가 같아도 **상수배**가 실무에서 중요한 이유를 안다
