## 왜 이 장인가

실무에서 가장 자주 쓰는 세 자료구조는 **배열**, **문자열**, **해시(Map/Set/Object)**다. 이 셋의 "연산 비용"을 감으로 알고 있어야 알고리즘 선택이 몸에 밴다.

이 장에서 집중적으로 풀 세 가지 질문.

1. 배열에서 `push`는 싸고 `unshift`는 왜 비싼가
2. 문자열이 **불변(immutable)**이라는 말이 실제로 무슨 뜻인가
3. `Object`, `Map`, `Set`은 무엇을 기준으로 골라야 하는가

## § 1 · 배열 — 연속된 메모리 덩어리

배열의 핵심은 **연속**이다. 모든 요소가 메모리에서 나란히 붙어 있고, `a[i]`는 "시작 주소 + i × 요소 크기"로 단번에 도달한다(O(1)).

<ArrayMemory />

이 구조가 그대로 연산 비용으로 드러난다.

| 연산 | 복잡도 | 이유 |
|---|---|---|
| `a[i]` 인덱싱 | **O(1)** | 주소 계산 한 번 |
| `push` / `pop` | 평균 **O(1)** | 끝에서 작업, 재배치 없음 |
| `unshift` / `shift` | **O(n)** | 전체를 한 칸 밀어야 함 |
| `splice(i, k)` | **O(n)** | 뒷부분 전체 재배치 |
| `slice(i, j)` | **O(k)** | 자를 길이만큼 복사 |
| `indexOf(x)` | **O(n)** | 선형 스캔 |

"첫 요소에 뭔가 자주 넣고 빼야 한다"면 배열이 아니라 **덱(Deque)** 구조를 써야 한다(다음 장).

## § 2 · 문자열 — 왜 불변인가

JS에서 문자열은 만든 뒤 바꿀 수 없다. `s += '!'`는 `s` 객체의 내용을 바꾸는 게 아니라 **새 문자열을 만들어 `s`에 재할당**한다.

```js
let s = 'abc';
s += 'd';
// 동작: 새 문자열 'abcd'를 만들고, s를 새 문자열로 교체
// 원래 'abc'는 가비지 컬렉션 대상
```

그래서 긴 문자열을 반복해 연결하면 매 단계마다 **전체 복사**가 일어난다 — `n`번 연결이 `O(n²)`이 된다.

```js
// 나쁨 — O(n²)
let out = '';
for (const x of items) out += x;

// 좋음 — O(n)
const parts = [];
for (const x of items) parts.push(x);
const out = parts.join('');
```

모던 엔진(V8)은 이런 패턴을 어느 정도 최적화하지만, **"불변이라는 사실"과 "배열 + join 패턴이 더 안전하다"**는 감각은 반드시 가지고 있어야 한다.

## § 3 · 투 포인터와 슬라이딩 윈도우

배열에서 "특정 조건을 만족하는 구간"을 찾는 문제는 거의 전부 두 패턴으로 풀린다.

### 투 포인터

**정렬된** 배열에서 두 원소의 합이 목표값이 되는 쌍 찾기.

```js
function twoSum(sorted, target) {
  let l = 0, r = sorted.length - 1;
  while (l < r) {
    const sum = sorted[l] + sorted[r];
    if (sum === target) return [l, r];
    if (sum < target) l++;      // 키운다
    else              r--;      // 줄인다
  }
  return null;
}
// 시간: O(n), 공간: O(1)
```

정렬이 이미 되어 있다면 이중 루프(O(n²))가 **O(n)**으로 내려간다.

### 슬라이딩 윈도우

연속 부분의 합이 `k`가 되는 구간 찾기.

```js
function windowSum(arr, k) {
  let sum = 0, start = 0, count = 0;
  for (let end = 0; end < arr.length; end++) {
    sum += arr[end];
    while (sum > k) sum -= arr[start++]; // 왼쪽 줄이기
    if (sum === k) count++;
  }
  return count;
}
```

"한 번 본 요소는 다시 보지 않는다"가 핵심. 각 요소가 윈도우에 **한 번 들어오고 한 번 나가므로** 총 O(n).

## § 5 · Object vs Map vs Set

실무 판단 기준:

| 기준 | Object | Map | Set |
|---|---|---|---|
| 키 타입 | 문자열/심볼만 | 모든 값 | 값만(중복 없음) |
| 삽입 순서 유지 | 부분적 | **완전 보장** | 완전 보장 |
| 크기 조회 | `Object.keys(o).length` — O(n) | `m.size` — **O(1)** | `s.size` — O(1) |
| 기본값(프로토타입) | 있음 — 오염 위험 | 없음 — 깨끗 | 없음 |
| JSON 직렬화 | 기본 지원 | 커스텀 필요 | 커스텀 필요 |
| 반복 | 별도 API 필요 | 내장 `for...of` | 내장 `for...of` |

정리하면:

- **JSON 입출력**이 목적이면 `Object`
- **키에 객체**가 들어가거나, **삽입 순서**가 중요하거나, **빈번한 크기 조회**가 있으면 `Map`
- **중복 판정 + 집합 연산**이면 `Set`

```js
// "이 사용자가 이미 본 글인가?"
const seen = new Set();
seen.add(postId);
seen.has(postId); // O(1)

// "글 ID → 읽은 시간"
const readAt = new Map();
readAt.set(postId, Date.now());
```

## § 6 · 체크포인트

- [ ] `a[i]`가 왜 O(1)인지 메모리 레이아웃으로 설명할 수 있다
- [ ] `unshift`가 O(n)인 이유를 그림으로 그릴 수 있다
- [ ] 반복 문자열 연결이 왜 O(n²)이 되는지 안다
- [ ] 투 포인터가 이중 루프를 어떻게 단일 순회로 바꾸는지 안다
- [ ] 해시 테이블이 최악 O(n)이 되는 조건을 안다
- [ ] Map과 Object를 어떤 기준으로 구분해 쓰는지 즉답할 수 있다
