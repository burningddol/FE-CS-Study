## 왜 이 장인가

`0.1 + 0.2`는 왜 `0.3`이 아닌가? 답은 수학이 아니라 **비트**에 있다. 컴퓨터는 모든 숫자를 켜짐/꺼짐 두 상태의 조합으로만 저장하는데, 10진수 `0.1`은 2진수로 표현하면 영원히 끝나지 않는 반복 소수다. 그래서 JS는 가장 가까운 2진수 근사치를 저장하고, 그 미세한 오차가 `+` 한 번 만으로 드러난다.

이 장에서 얻을 것은 세 가지다.

1. **정수**가 내부에서 어떻게 저장되고, 음수가 왜 굳이 "2의 보수" 방식을 쓰는가
2. **실수**가 왜 오차를 피할 수 없고, `Number.EPSILON`이 왜 필요한가
3. **비트 연산자**가 날것으로는 난해해 보이지만, 권한 플래그·색상·성능 트릭에서 어떻게 이 연산이 한 줄 코드로 변하는가

## § 1 · 2진수·16진수와 자릿값

10진수에서 `213`은 `2·10² + 1·10¹ + 3·10⁰`이다. 2진수도 원리는 같다 — 밑이 10 대신 2로 바뀔 뿐이다.

<BinaryBits value={13} />

0과 1만 쓰기 때문에 비트 수가 금방 길어지고, 그래서 사람이 읽을 때는 **16진수**로 줄여 쓴다. 4비트가 16진수 한 자리에 정확히 대응하기 때문에, `1101 1001` 같은 2진수는 `0xD9`로 두 자리에 들어간다. 색상 코드(`#ff0088`), 메모리 주소(`0x7ffee0…`)가 16진수로 쓰이는 이유다.

```js
(0xff).toString(2);     // "11111111"
(0b10101010).toString(16); // "aa"
parseInt("1d9", 16);    // 473
```

## § 2 · 음수는 어떻게? — 2의 보수

가장 왼쪽 비트를 "부호 전용"으로 쓰는 방식(부호-크기 표현)은 간단해 보이지만 두 가지 큰 문제를 만든다: `+0`과 `-0`이 따로 존재하고, 덧셈 회로에서 양수/음수 분기 로직이 필요해진다.

**2의 보수**는 이 문제를 구조적으로 해결한다. `x`의 2의 보수는 "모든 비트를 뒤집고 1을 더한 값"으로 정의된다. 4비트에서 `+3 = 0011`, `-3 = 1101`이 되는데, 이 둘을 2진수 그대로 더하면 `0011 + 1101 = 10000` → 4비트로 잘라내면 `0000`. 즉 **음수와 양수가 자연스럽게 상쇄**된다. 별도의 빼기 회로가 필요 없다.

<TwosComplement />

원이 닫히기 때문에 가장자리에서 **오버플로우**가 일어난다. 4비트 signed의 최댓값 `+7`에 `1`을 더하면 `1000` = `-8`. JS에서도 비트 연산은 내부적으로 32-bit signed integer로 변환되므로 이 감각이 그대로 적용된다.

```js
(2 ** 31 - 1) | 0;        // 2147483647  (INT32_MAX)
((2 ** 31 - 1) + 1) | 0;  // -2147483648 (오버플로우)
```

## § 3 · IEEE 754 — 실수의 내부

정수와 달리 실수는 64비트를 셋으로 쪼개 저장한다.

<IEEE754 />

`0.1`을 2진수로 쓰면 `0.0001100110011…`이 무한 반복된다. 가수 영역은 52비트뿐이니 잘릴 수밖에 없다. 그 잘린 오차가 `0.1 + 0.2`를 `0.30000000000000004`로 만든다.

```js
0.1 + 0.2 === 0.3; // false
0.1 + 0.2;         // 0.30000000000000004
```

비교할 때는 등호 대신 **오차 허용 범위**를 쓴다.

```js
Math.abs(0.1 + 0.2 - 0.3) < Number.EPSILON; // true
```

정수도 안전 범위가 있다. `Number.MAX_SAFE_INTEGER = 2^53 - 1`. 이 이상은 정밀도가 깨지므로 `BigInt`를 쓴다.

```js
Number.MAX_SAFE_INTEGER;     // 9007199254740991
9007199254740993;            // 9007199254740992 (틀림)
9007199254740993n;           // BigInt로는 정확
```

## § 4 · 비트 연산자의 문법

| 연산자 | 이름 | 의미 |
|---|---|---|
| `&` | AND | 양쪽 다 1일 때만 1 |
| <code>\|</code> | OR | 하나라도 1이면 1 |
| `^` | XOR | 다를 때만 1 |
| `~` | NOT | 비트 전체 반전 |
| `<<` | 좌 시프트 | 왼쪽으로 밀기 (×2) |
| `>>` | 산술 우 시프트 | 부호 비트 유지 |
| `>>>` | 논리 우 시프트 | 0으로 채움 |

알고 있으면 아래 패턴들이 한 줄이 된다.

```js
// 짝/홀 판정
n & 1;                    // 0 → 짝, 1 → 홀

// 2의 거듭제곱인가?
(n & (n - 1)) === 0;      // n = 16 → 10000 & 01111 = 0 → true

// 특정 비트 켜기 / 끄기 / 토글
flags |  BIT;             // set
flags & ~BIT;             // clear
flags ^  BIT;             // toggle
```

## § 5 · FE 실전에서 어디에 쓰이나

### 권한 비트마스크 (RBAC)

문자열 배열보다 비트 한 덩어리가 훨씬 가볍고 빠르다.

```js
const READ    = 1 << 0; // 0b0001
const WRITE   = 1 << 1; // 0b0010
const EXECUTE = 1 << 2; // 0b0100

let perms = READ | WRITE;          // 0b0011
const canWrite = (perms & WRITE) !== 0; // true

perms &= ~WRITE;                   // 권한 취소
```

### 색상 RGB 합성

```js
const r = 0xff, g = 0x88, b = 0x00;
const color = (r << 16) | (g << 8) | b;
color.toString(16); // "ff8800"
```

### 성능 트릭

```js
n << 1;     // n * 2
n >> 1;     // Math.floor(n / 2)  (signed만)
~~x;        // Math.trunc(x)에 가까움 — 32-bit로 잘림
x | 0;      // 정수화
```

단, 이런 트릭은 가독성 비용이 크다. 성능이 정말로 핫한 구간에서만 쓴다.

## § 6 · 체크포인트

- [ ] 2의 보수가 왜 "부호 + 절댓값" 방식보다 **회로적으로** 유리한지 설명할 수 있다
- [ ] `0.1 + 0.2 !== 0.3`의 원인을 IEEE 754의 가수 비트 개수로 설명할 수 있다
- [ ] `Number.MAX_SAFE_INTEGER`가 `2^53 - 1`인 이유를 안다
- [ ] `(n & (n - 1)) === 0`이 왜 2의 거듭제곱 판정이 되는지 비트 그림으로 그릴 수 있다
- [ ] 비트 시프트가 JS에서 내부적으로 32-bit signed로 변환된다는 사실을 안다
