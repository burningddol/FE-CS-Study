## 왜 이 장인가

이산수학은 "연속적이지 않은" 수학이다. 알고리즘이 다루는 대상 — 배열의 원소, 트리의 노드, 조건문의 참/거짓 — 은 모두 셀 수 있는 덩어리다. 그래서 **이산수학의 언어가 곧 알고리즘의 언어**가 된다.

특히 세 가지가 FE 실무에 직접 닿는다.

1. **명제 논리** — 복잡한 `if` 조건을 수학적으로 리팩토링하는 근거
2. **집합·모듈러** — 해시와 중복 제거가 왜 빠른지에 대한 기초
3. **귀납법** — 재귀가 올바른지 **증명**하는 방식

## § 1 · 명제 논리와 드모르간 법칙

두 명제 `A`, `B`가 있을 때 다음 두 식은 **항상** 동치다.

```
¬(A ∧ B)  ≡  (¬A ∨ ¬B)
¬(A ∨ B)  ≡  (¬A ∧ ¬B)
```

자연어로: "둘 다 참은 아니다" = "적어도 하나는 거짓이다". 이 한 줄이 조건문 리팩토링의 전부다.

```js
// before
if (!(user.isAdmin && user.hasSession)) redirect();

// after — 드모르간 적용
if (!user.isAdmin || !user.hasSession) redirect();
```

> 두 형태는 **논리적으로 동치**이지만 가독성은 전혀 다르다. 팀이 "거부 조건"을 먼저 읽고 싶어하는지, "허용 조건"을 먼저 읽고 싶어하는지에 따라 선택이 갈린다.

## § 2 · 집합과 배열 메서드

집합 연산은 배열로도 흉내 낼 수 있지만, 복잡도가 달라진다.

| 연산 | 배열 기반 | `Set` 기반 |
|---|---|---|
| 합집합 | `[...a, ...b]`로 만들고 중복 제거 → O(n²) | `new Set([...a, ...b])` → **O(n)** |
| 교집합 | `a.filter(x => b.includes(x))` → O(n·m) | `a.filter(x => bSet.has(x))` → **O(n)** |
| 차집합 | 같은 이유로 O(n·m) | `a.filter(x => !bSet.has(x))` → **O(n)** |

`Array.includes`는 내부적으로 O(n) 선형 검색이고, `Set.has`는 평균 O(1) 해시 조회다. **한쪽을 Set으로 바꾸기만 해도 체감 성능이 수백 배 차이**가 난다.

```js
// 느림 — O(n·m)
a.filter(x => b.includes(x));

// 빠름 — O(n)
const bSet = new Set(b);
a.filter(x => bSet.has(x));
```

## § 3 · 순열·조합 — 경우의 수

- **순열** `nPr`: 순서를 고려해 n개 중 r개 뽑기 — `n! / (n-r)!`
- **조합** `nCr`: 순서 무시 — `n! / (r! · (n-r)!)`
- **중복 순열**: 같은 원소 재사용 허용 — `n^r`

백트래킹 문제는 거의 전부 이 세 식 중 하나로 환원된다. "경우의 수가 얼마인가?"를 먼저 계산하면 **시간 내에 풀 수 있는지**를 먼저 판정할 수 있다. 10명 중 5명을 순서대로 뽑는 건 30,240가지지만, 20명이라면 1.8억 가지로 폭증한다.

## § 4 · 모듈러 연산

`a mod n`은 "`a`를 `n`으로 나눈 나머지"다. 해시 테이블이 `hash(key) % bucketCount`로 버킷을 정하는 이유는, 나머지 연산이 값을 **항상 `[0, n)` 범위 안에 가둬주기** 때문이다.

JS의 `%`에는 함정이 있다.

```js
-1 % 5;   // -1 (음수 부호 유지)
```

수학적 정의대로라면 `4`가 되어야 하지만 JS는 부호를 유지한다. 음수 인덱스를 안전히 감싸려면 `((n % m) + m) % m`을 쓴다.

```js
const safeMod = (n, m) => ((n % m) + m) % m;
safeMod(-1, 5); // 4
```

## § 5 · 수학적 귀납법 — 재귀의 증명

귀납법은 두 단계로 이뤄진다.

1. **기저 사례**: 가장 작은 입력에서 명제가 참
2. **귀납 단계**: `k`일 때 참이라고 가정하고, `k+1`일 때도 참임을 보임

재귀 함수가 올바른지 확신이 들지 않을 때 이 틀을 그대로 적용한다.

```js
// sum(n) = 1 + 2 + ... + n = n(n+1)/2 인가?
function sum(n) {
  if (n === 0) return 0;     // 기저
  return n + sum(n - 1);      // 귀납
}
```

- 기저: `sum(0) = 0 = 0·1/2` ✓
- 귀납: `sum(k) = k(k+1)/2`라 가정하면, `sum(k+1) = (k+1) + k(k+1)/2 = (k+1)(k+2)/2` ✓

반복문에서는 같은 개념이 **불변량(loop invariant)**으로 나타난다. "이 루프가 한 번 돌 때마다 이 조건은 항상 참이다"를 명시할 수 있으면 반복 로직을 증명할 수 있다.

## § 6 · 체크포인트

- [ ] 드모르간을 자연어로 설명할 수 있다
- [ ] `Array.includes`를 `Set.has`로 바꿀 때 복잡도가 어떻게 달라지는지 안다
- [ ] JS의 음수 모듈러가 왜 수학적 정의와 다른지 안다
- [ ] 재귀 함수의 정당성을 기저·귀납 2단계로 쪼개 설명할 수 있다
- [ ] 반복문 불변량이 무엇인지 안다
