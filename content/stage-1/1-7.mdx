## 왜 이 장인가

앞 장들이 **자료구조**였다면, 이 장은 그 위에서 동작하는 **알고리즘 패턴**이다. 실전에서 맞닥뜨리는 문제는 대부분 아래 8가지 패턴 중 하나로 환원된다.

정렬 · 이분 탐색 · 분할 정복 · DP · 백트래킹 · 그리디 · 위상 정렬 · 최단 경로.

이 패턴들의 **판별법**을 익히는 것이 이 장의 목표다. "이 문제는 DP일까 그리디일까"를 감으로 구분할 수 있으면 문제의 반은 푼 셈이다.

## § 1 · 정렬 — 셋만 기억하면 된다

### 개념 수준만 알고 있으면 되는 것

| 알고리즘 | 복잡도 | 메모 |
|---|---|---|
| 버블 | O(n²) | 교육용 |
| 선택 | O(n²) | 비교는 많지만 swap은 적음 |
| 삽입 | O(n²) 최악 / **O(n) 거의 정렬됨** | Timsort의 부품 |

### 실전에서 중요한 것

**퀵 정렬** — 평균 O(n log n), 최악 O(n²). **in-place**(추가 공간 O(1)). 랜덤 피벗이면 사실상 항상 O(n log n).

**머지 정렬** — 항상 O(n log n) 보장. **stable**(같은 값의 상대 순서 유지). 추가 공간 O(n).

**힙 정렬** — 항상 O(n log n), in-place, 하지만 **unstable**.

### JS `Array.sort()`가 실제로 쓰는 것

**Timsort** — 머지 + 삽입의 하이브리드. "거의 정렬된 구간(run)"을 먼저 찾아내 그 안에서는 삽입 정렬(빠름), 큰 단위에서는 머지(안정적). **stable**이며, ES2019부터 모든 엔진에서 stable 보장.

```js
// 객체 정렬에서 stable이 왜 중요한가?
const items = [
  { name: 'A', priority: 2 },
  { name: 'B', priority: 1 },
  { name: 'C', priority: 2 }, // A와 같은 priority
];
items.sort((a, b) => a.priority - b.priority);
// stable이면 A, C의 상대 순서가 보장된다 — 중요한 UX
```

## § 2 · 이분 탐색

**정렬된** 배열에서 `O(log n)`. 창을 절반씩 자른다.

```js
function binarySearch(arr, target) {
  let lo = 0, hi = arr.length - 1;
  while (lo <= hi) {
    const mid = (lo + hi) >> 1;
    if (arr[mid] === target) return mid;
    if (arr[mid] < target) lo = mid + 1;
    else                   hi = mid - 1;
  }
  return -1;
}
```

### `lower_bound` — 삽입 위치 찾기

"이 값을 넣어도 정렬이 깨지지 않는 가장 왼쪽 위치".

```js
function lowerBound(arr, target) {
  let lo = 0, hi = arr.length;
  while (lo < hi) {
    const mid = (lo + hi) >> 1;
    if (arr[mid] < target) lo = mid + 1;
    else                   hi = mid;
  }
  return lo;
}
```

### 파라메트릭 서치 — 답을 이분 탐색

"최댓값을 구하라" 유형에서, 답의 범위가 정해져 있고 "이 값이 조건을 만족하는가?"를 판정할 수 있으면 **답 자체를 이분 탐색**한다.

```js
// 길이 n의 로프를 k개의 자식에게 같게 나눠줄 수 있는 최대 길이?
// 답의 범위 [1, max(ropes)]에서 "길이 L이 가능한가?"를 이분 탐색
```

## § 3 · 재귀와 반복 — 경계 조건

재귀는 우아하지만 함정이 있다.

- **JS의 재귀 깊이 제한** — 스택 오버플로우 전까지 약 1만 프레임. 깊은 DFS는 재귀 대신 **명시적 스택** 사용.
- **꼬리 재귀 최적화(TCO)** — ES6 스펙에는 있지만 V8 등 대부분의 엔진이 **구현 안 함**. JS에서는 기대하지 말 것.
- **중복 호출** — 메모이제이션 없이 재귀 피보나치를 짜면 `O(2ⁿ)`으로 폭발한다.

## § 4 · 백트래킹 — 가지치기가 전부

"모든 경우를 시도하되, **틀린 가지는 조기에 버린다**".

```js
// N-Queens — n × n 판에 n개 퀸을 서로 공격하지 않게 놓기
function nQueens(n) {
  const cols = new Set(), diag1 = new Set(), diag2 = new Set();
  let count = 0;
  function backtrack(row) {
    if (row === n) { count++; return; }
    for (let col = 0; col < n; col++) {
      if (cols.has(col) || diag1.has(row - col) || diag2.has(row + col)) continue;
      cols.add(col); diag1.add(row - col); diag2.add(row + col);
      backtrack(row + 1);
      cols.delete(col); diag1.delete(row - col); diag2.delete(row + col);
    }
  }
  backtrack(0);
  return count;
}
```

핵심은 **"이 선택이 불가능하면 즉시 되돌아간다"**는 pruning. 가지치기가 없으면 단순 완전 탐색과 같다.

## § 5 · 분할 정복

"큰 문제를 같은 형태의 작은 문제 여러 개로 쪼갠 뒤, 해결한 것을 합친다". 머지 정렬·퀵 정렬이 정확히 이 패턴.

점화식이 `T(n) = a·T(n/b) + f(n)` 꼴이면 **마스터 정리**로 복잡도를 바로 구할 수 있다.

## § 6 · 동적 계획법 (DP) — 겹치는 부분 문제

DP의 두 조건:

1. **최적 부분 구조**: 큰 문제의 최적해가 작은 부분 문제의 최적해로부터 만들어진다
2. **중복되는 부분 문제**: 같은 부분 문제가 여러 번 나타난다

### Top-down (메모이제이션)

```js
function fib(n, memo = {}) {
  if (n < 2) return n;
  if (memo[n] !== undefined) return memo[n];
  return memo[n] = fib(n - 1, memo) + fib(n - 2, memo);
}
```

### Bottom-up (타뷸레이션)

```js
function fib(n) {
  if (n < 2) return n;
  const dp = [0, 1];
  for (let i = 2; i <= n; i++) dp[i] = dp[i - 1] + dp[i - 2];
  return dp[n];
}
```

현실적으론 공간을 더 줄여 **이전 두 값만 유지**하면 O(1) 공간에 해결된다.

### 대표 유형

- **LIS** (Longest Increasing Subsequence) — 가장 긴 증가 부분수열
- **LCS** (Longest Common Subsequence) — 두 수열의 가장 긴 공통 부분수열
- **배낭 문제** — 무게 제한 하에서 가치 최대화
- **동전 교환** — 최소 동전 수
- **편집 거리** — 아래 § 10 참고

## § 7 · 그리디 — 증명이 없으면 틀린다

매 단계에서 "지금 시점에서 가장 좋은 선택"을 한다. 빠르지만 **정당성 증명**이 필수다.

고전 예: 동전 교환.

```
동전: [1, 5, 10, 50, 100, 500]
거스름돈 760원을 최소 동전 수로 만들려면?
→ 그리디(큰 동전부터): 500 + 100 + 100 + 50 + 10 = 5개 ✓
```

하지만 동전이 `[1, 4, 5]`이고 8원을 만든다면:

```
그리디: 5 + 1 + 1 + 1 = 4개
최적  : 4 + 4          = 2개  ← 그리디가 틀림
```

**교환 논증**으로 그리디의 옳음을 증명할 수 있어야 한다. 증명이 안 되면 DP를 써야 한다.

## § 8 · 위상 정렬 — 번들러가 쓰는 바로 그것

**DAG**(방향 있는 비순환 그래프)에서 "어떤 순서로 처리하면 선후 관계가 지켜지는가"를 정한다. 모듈 의존성 해결의 기반.

### Kahn의 알고리즘 — 진입 차수 기반

```js
function topoSort(graph) {
  const inDegree = {};
  for (const node in graph) inDegree[node] ??= 0;
  for (const node in graph) {
    for (const next of graph[node]) inDegree[next] = (inDegree[next] ?? 0) + 1;
  }

  const queue = Object.keys(inDegree).filter(n => inDegree[n] === 0);
  const result = [];
  while (queue.length) {
    const node = queue.shift();
    result.push(node);
    for (const next of graph[node] ?? []) {
      inDegree[next]--;
      if (inDegree[next] === 0) queue.push(next);
    }
  }

  return result.length === Object.keys(inDegree).length ? result : null; // null = 순환
}
```

번들러는 위상 정렬로 **"누가 먼저 로드되어야 하는가"**를 결정한다. `import` 순환(circular dependency)은 위상 정렬이 실패할 때 튀어나오는 에러다.

## § 9 · 최단 경로

| 알고리즘 | 조건 | 복잡도 |
|---|---|---|
| **BFS** | 가중치 없음 | O(V + E) |
| **다익스트라** | 가중치 ≥ 0 | O((V + E) log V) with heap |
| **벨만-포드** | 음수 가중치 허용 | O(V · E) |
| **플로이드-워셜** | 모든 쌍 | O(V³) |

다익스트라는 사실상 "**가중치 있는 BFS**"다. 큐 대신 우선순위 큐(힙)를 쓰고, "지금까지 가장 가까운 노드"를 매 단계 꺼낸다. FE에서 라우팅·경로 계산을 직접 구현할 일은 드물지만, **네트워크 라우팅**이 이 알고리즘 위에 돌아간다.

## § 10 · 편집 거리 — DP의 왕

두 문자열을 같게 만드는 **최소 삽입/삭제/교체 연산 수**. diff, 검색 유사도, 오타 제안의 기반.

```
       ""  k   i   t   t   e   n
   ""   0  1   2   3   4   5   6
   s    1  1   2   3   4   5   6
   i    2  2   1   2   3   4   5
   t    3  3   2   1   2   3   4
   t    4  4   3   2   1   2   3
   i    5  5   4   3   2   2   3
   n    6  6   5   4   3   3   2
   g    7  7   6   5   4   4   3   ← 답: 3
```

```js
function editDistance(a, b) {
  const m = a.length, n = b.length;
  const dp = Array.from({ length: m + 1 }, () => Array(n + 1).fill(0));
  for (let i = 0; i <= m; i++) dp[i][0] = i;
  for (let j = 0; j <= n; j++) dp[0][j] = j;
  for (let i = 1; i <= m; i++) {
    for (let j = 1; j <= n; j++) {
      if (a[i - 1] === b[j - 1]) dp[i][j] = dp[i - 1][j - 1];
      else dp[i][j] = 1 + Math.min(dp[i - 1][j], dp[i][j - 1], dp[i - 1][j - 1]);
    }
  }
  return dp[m][n];
}
```

**Virtual DOM diff가 왜 이 문제의 단순화 버전인가**: 두 트리의 "최소 편집 연산"을 구하는 것이 원래 문제이나, 풀면 O(n³) 이상이라 React는 발견적 알고리즘(heuristic)으로 **같은 레벨의 같은 타입 노드만 비교**하는 방식으로 O(n)으로 낮춘다.

## § 11 · 패턴 판별 — 이 문제는 어떤 패턴인가?

| 문제가 이렇게 생겼으면 | 패턴 |
|---|---|
| "정렬된 배열에서 x를 찾아라" | 이분 탐색 |
| "모든 경우의 수를 구하라" / "가능한 조합을 전부" | 백트래킹 |
| "최소/최대 ~을 구하라" + 부분 문제 중복 | DP |
| "최소/최대 ~을 구하라" + 탐욕이 증명 가능 | 그리디 |
| "선후 관계가 있는 작업들의 순서" | 위상 정렬 |
| "가중치 0인 최단 거리" | BFS |
| "가중치 양수인 최단 거리" | 다익스트라 |
| "두 문자열의 유사도" | 편집 거리 (DP) |

## § 12 · 체크포인트

- [ ] Timsort가 왜 stable인 것이 실무에서 의미가 있는지 설명할 수 있다
- [ ] 이분 탐색에서 `lo <= hi` vs `lo < hi`의 경계 조건을 구분할 수 있다
- [ ] 백트래킹이 완전 탐색과 어떻게 다른지 (가지치기) 설명할 수 있다
- [ ] DP의 두 조건(최적 부분 구조, 중복 부분 문제)을 외우고 있다
- [ ] 그리디가 정당성 증명 없이는 믿을 수 없는 이유를 **반례**로 보일 수 있다
- [ ] Kahn 알고리즘의 진입 차수 갱신 순서를 설명할 수 있다
- [ ] 다익스트라가 왜 우선순위 큐를 쓰는지(가장 가까운 노드부터 확정) 안다
- [ ] 편집 거리 DP 테이블의 점화식을 칠판에 그릴 수 있다
