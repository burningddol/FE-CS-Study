## 왜 이 장인가

연결 리스트·트리·그래프는 모두 **노드와 포인터**라는 한 가지 아이디어의 변형이다. 노드가 옆 노드 하나만 가리키면 연결 리스트, 자식 둘을 가리키면 이진 트리, 아무 노드나 가리킬 수 있으면 그래프.

**FE 실무는 이 구조가 핵심이다** — DOM 트리, Virtual DOM, 컴포넌트 트리, 번들러의 의존성 그래프. 모두 노드-포인터 자료구조다.

## § 1 · 연결 리스트

각 노드가 **값 + 다음 노드 포인터**를 가진다.

### 배열과의 근본적 차이

| 기준 | 배열 | 연결 리스트 |
|---|---|---|
| 메모리 | 연속 | 분산 |
| `a[i]` 접근 | **O(1)** | O(n) — 처음부터 따라가야 |
| 중간 삽입/삭제 | O(n) — 재배치 | **O(1)** — 포인터만 바꿈 |
| 캐시 효율 | 좋음 | 나쁨 |

배열이 압도적으로 유리한 경우가 대부분이지만, **"중간에 빈번히 삽입·삭제"**가 있고 **"인덱스 접근"**이 필요 없을 때 연결 리스트가 선택된다.

### 대표 패턴 — 뒤집기

```js
function reverse(head) {
  let prev = null, curr = head;
  while (curr) {
    const next = curr.next;
    curr.next = prev;
    prev = curr;
    curr = next;
  }
  return prev;
}
```

포인터 세 개(`prev`, `curr`, `next`)만 기억하면 된다.

## § 2 · 이진 트리와 BST

각 노드가 **최대 두 자식**을 가지면 이진 트리. **BST(이진 탐색 트리)**는 여기에 추가 규칙 — 왼쪽 서브트리는 모두 작고, 오른쪽은 모두 크다.

### 순회 4가지

- **전위(preorder)**: 부모 → 왼쪽 → 오른쪽 — 트리 복제, 표현식 변환
- **중위(inorder)**: 왼쪽 → 부모 → 오른쪽 — **BST에선 정렬된 순서로 나옴**
- **후위(postorder)**: 왼쪽 → 오른쪽 → 부모 — 트리 삭제, 디렉토리 크기 계산
- **레벨(BFS)**: 위에서 아래 한 층씩 — 가장 얕은 노드 찾기, UI 트리 깊이 제한

### BST의 함정

균형이 무너진 BST는 **연결 리스트와 다름없다**. 모든 요소를 순서대로 삽입하면 높이가 `n`이 되고 탐색은 O(n).

```js
// 삽입 순서: 1, 2, 3, 4, 5
//   1
//    \
//     2
//      \
//       3  ← 사실상 linked list
```

그래서 실무에서는 **AVL 트리**, **Red-Black 트리** 같은 자동 균형 BST를 쓴다 (대부분의 언어의 TreeMap/TreeSet 내부).

## § 3 · 트라이 (Trie)

문자열을 **문자 단위로 분기**하는 트리.

```
                  ∅
                 / \
                a   c
                |   |
                p   a
               /|   |
              e t   t
              |
              r
              |
              •  ← 여기까지가 "aper"
```

**접두사 검색**이 핵심. 한 번 구축하면 "ap로 시작하는 모든 단어"를 문자열 길이 `m`에 비례하는 시간에 찾는다. 주소 자동완성, 이모지 검색, 태그 필터의 내부.

```js
class Trie {
  root = {};
  insert(word) {
    let node = this.root;
    for (const ch of word) node = (node[ch] ??= {});
    node.end = true;
  }
  hasPrefix(prefix) {
    let node = this.root;
    for (const ch of prefix) {
      if (!node[ch]) return false;
      node = node[ch];
    }
    return true;
  }
}
```

**삽입·검색 모두 O(m)** — 전체 단어 개수와 무관하게 검색어 길이에만 비례한다.

## § 4 · Union-Find (서로소 집합)

"이 두 노드는 같은 그룹인가?"를 O(≈1)에 답하는 자료구조. 각 원소가 자기 집합의 대표자(루트)를 가리킨다.

```js
class UnionFind {
  parent = [];
  rank = [];
  constructor(n) {
    this.parent = Array.from({ length: n }, (_, i) => i);
    this.rank = Array(n).fill(0);
  }
  find(x) {
    if (this.parent[x] !== x) {
      this.parent[x] = this.find(this.parent[x]); // 경로 압축
    }
    return this.parent[x];
  }
  union(a, b) {
    const ra = this.find(a), rb = this.find(b);
    if (ra === rb) return;
    if (this.rank[ra] < this.rank[rb]) this.parent[ra] = rb;
    else if (this.rank[ra] > this.rank[rb]) this.parent[rb] = ra;
    else { this.parent[rb] = ra; this.rank[ra]++; }     // rank by rank
  }
}
```

**경로 압축** + **Union by Rank** 두 최적화를 합치면 연산당 거의 O(1)(정확히는 역 아커만 함수 `α(n)`). MST(Kruskal 알고리즘)의 핵심.

## § 5 · 그래프 — 표현과 순회

### 인접 행렬 vs 인접 리스트

| 표현 | 공간 | 간선 확인 | 모든 이웃 탐색 |
|---|---|---|---|
| 인접 행렬 | **O(V²)** | O(1) | O(V) |
| 인접 리스트 | **O(V + E)** | O(차수) | O(차수) |

그래프가 희소(간선이 적음)할수록 인접 리스트가 유리하다. FE에서 만나는 대부분의 그래프(DOM 트리, 번들러 의존성)는 희소하다.

```js
// 인접 리스트
const graph = {
  A: ['B', 'C'],
  B: ['D'],
  C: ['D', 'E'],
  D: [],
  E: ['A'],  // 순환
};
```

### DFS vs BFS

**DFS** — 스택(또는 재귀). 한 방향으로 끝까지 파고든 뒤 돌아온다.

```js
function dfs(graph, start) {
  const visited = new Set();
  const stack = [start];
  while (stack.length) {
    const node = stack.pop();
    if (visited.has(node)) continue;
    visited.add(node);
    for (const next of graph[node]) stack.push(next);
  }
}
```

**BFS** — 큐. 시작점에서 가까운 노드부터 순서대로.

```js
function bfs(graph, start) {
  const visited = new Set([start]);
  const queue = [start];
  while (queue.length) {
    const node = queue.shift();
    for (const next of graph[node]) {
      if (!visited.has(next)) {
        visited.add(next);
        queue.push(next);
      }
    }
  }
}
```

**가중치 없는 최단 경로는 무조건 BFS**. 먼저 도달한 순서가 곧 짧은 순서다.

## § 6 · FE에서 이 구조들이 어디에 있는가

- **DOM 트리** — 브라우저가 HTML을 파싱해 만드는 트리. `document.body`에서 루트, 각 요소가 `childNodes`로 자식을 가진다.
- **Virtual DOM diff** — 두 트리를 비교해 바뀐 부분만 찾아내는 알고리즘. 깊이 순회 + 휴리스틱.
- **컴포넌트 트리** — React의 내부 표현. Fiber는 연결 리스트 + 트리의 혼합.
- **번들러 의존성 그래프** — webpack/vite가 `import` 관계를 그래프로 만든 뒤 위상 정렬로 빌드 순서를 정한다.

"트리/그래프를 안다"는 건 실무의 이런 장면들이 **같은 알고리즘 언어로 보인다**는 뜻이다.

## § 7 · 체크포인트

- [ ] 연결 리스트 뒤집기를 포인터 세 개로 구현할 수 있다
- [ ] BST의 중위 순회가 정렬된 순서를 내는 이유를 안다
- [ ] BST가 연결 리스트처럼 퇴화하는 조건을 안다
- [ ] 트라이가 접두사 검색 `O(m)`을 보장하는 이유를 안다
- [ ] Union-Find의 두 가지 최적화(경로 압축, Union by Rank)를 설명할 수 있다
- [ ] 인접 행렬과 인접 리스트를 **언제** 구분해 쓰는지 안다
- [ ] 가중치 없는 최단 경로가 왜 BFS인지 설명할 수 있다
